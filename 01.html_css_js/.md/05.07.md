# [05.07] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

웹퍼블리싱은 단순히 화면에 보이게 만드는 작업이 아니라, 브라우저가 이해할 수 있는 정보 구조를 만들고 그 구조가 사용자에게 렌더링되도록 조정하는 작업이다.

HTML은 정보 구조를 만들고, CSS는 그 구조가 만들어낸 박스와 텍스트를 시각적으로 표현하며, JavaScript는 이미 만들어진 구조를 동적으로 조작하거나 기본 동작을 제어한다.

강의의 주요 관점은 다음과 같다.

- 프로그래밍 언어와 웹 표준은 자연법칙이 아니라 사람이 정한 약속이다.
- 외워야 할 신비한 규칙이라기보다, 문서와 도구를 통해 확인하며 쓰는 규칙이다.
- HTML/CSS를 잘하려면 브라우저 렌더링 엔진의 사고방식을 이해해야 한다.
- AI로 코드를 무작정 생성하기보다, 왜 그렇게 동작하는지 검토하는 도구로 쓰는 것이 좋다.

## 2. HTML의 기본 구조와 `doctype`

HTML의 초창기 목적은 누구나 정보를 접근하고 공유할 수 있게 하는 것이었기에, 다소 관대하게 동작한다. 코드가 완전하지 않아도 브라우저가 내부적으로 필요한 구조를 보정해서 렌더링하기 때문이다.

이 관대함은 편리하지만, 실무에서는 위험할 수 있다. 브라우저가 보정한 결과가 개발자의 의도와 다를 수 있고, 브라우저마다 보정 방식의 차이가 문제를 만들 수 있기 때문이다.

`<!doctype html>` 선언은 브라우저에게 최신 표준 모드로 렌더링하라고 알리는 역할을 한다. 없으면 브라우저가 오래된 페이지 호환을 위한 Quirks Mode로 동작할 수 있다.

> Quirks Mode: 오래된 웹 페이지와의 하위 호환성을 유지하기 위해 표준과 다른 방식으로 렌더링하는 모드

핵심은 `doctype`을 단순한 주문처럼 외우는 것이 아니라, 브라우저의 렌더링 모드를 결정하는 신호로 이해하는 것이다.

## 3. 태그는 정보 구조를 만든다

HTML에서 태그 하나를 적는다는 것은 브라우저에게 정보 덩어리 하나를 만들라고 명령하는 것과 같다. 브라우저는 HTML을 읽으면서 토큰으로 쪼개고, 이를 기반으로 DOM 객체와 노드를 만든다.

강의에서는 태그를 단순한 글자 장식이 아니라 자료 구조를 설계하는 도구로 보라고 설명했다.

예를 들어 `div`와 `section`이 화면에서 비슷해 보일 수는 있다. 하지만 브라우저, 검색 엔진, 스크린 리더가 문서를 해석할 때는 태그의 의미가 단서가 된다.

따라서 HTML을 작성할 때는 "어떤 모양이 필요한가"보다 "이 정보가 문서 안에서 어떤 역할인가"를 먼저 판단하는 것이 좋다.

## 4. 브라우저 렌더링 흐름

브라우저는 HTML을 한 번에 끝까지 읽고 나서 처리하는 것이 아니라, 위에서 아래로 읽으며 토큰화하고 요소를 만들고 렌더링한다.

대략적인 흐름은 다음과 같다.

1. HTML 텍스트를 읽는다.
2. 시작 태그, 종료 태그, 텍스트 등을 토큰으로 나눈다.
3. 토큰을 기반으로 DOM 노드를 만든다.
4. CSS와 결합해 화면에 그릴 구조를 계산한다.
5. JavaScript가 중간에 있으면 실행이 끝날 때까지 아래쪽 처리가 막힐 수 있다.

그래서 JavaScript는 보통 문서 구조가 만들어진 뒤 실행되도록 배치하는 것이 좋다. 전통적으로는 `body` 끝에 두거나, 현대적으로는 `defer`를 사용해 HTML 파싱 이후 실행되게 만든다.

JavaScript를 문서 중간에 무심코 넣으면 렌더링이 멈추거나 사용자가 화면이 버벅인다고 느낄 수 있다. 이 문제는 단순 문법 문제가 아니라 브라우저가 HTML을 읽고 실행하는 순서의 문제다.

## 5. HTML과 SEO, 시맨틱 웹

HTML 태그는 화면 모양만 결정하지 않는다. 검색 엔진, 스크린 리더, 브라우저, 크롤러가 문서의 의미를 이해하는 단서가 된다.

예를 들어 모든 내용을 `p`로만 작성하면 검색 엔진 입장에서는 정보의 위계가 약하다. 반대로 `h1`, `h2`, `article`, `section`, `em`, `strong` 같은 태그를 문맥에 맞게 쓰면 문서의 의미와 강약이 더 잘 드러난다.

이를 시맨틱 웹이라고 볼 수 있다. 사람이 보는 화면뿐 아니라 기계가 읽어도 의미가 드러나도록 마크업하는 것이다.

실무에서는 시맨틱 태그를 쓰는 이유를 "SEO에 좋다" 정도로만 이해하면 부족하다. 접근성, 유지보수, 자동화 도구, 검색 노출, AI 기반 분석까지 모두 문서 구조의 영향을 받는다.

## 6. `h1`과 제목 구조

`h1`은 한 문서에서 하나만 사용하는 것이 실무적으로 안전하다. 문법적으로 여러 번 써도 브라우저가 바로 에러를 내지는 않지만, SEO와 문서 위계 관점에서는 대표 제목을 하나로 두는 편이 명확하다.

제목 태그는 글자 크기를 정하기 위한 태그가 아니라 의미와 위계를 표현하기 위한 태그다. 시각적 크기는 CSS로 조절한다.

실무적으로는 다음 원칙이 안전하다.

- 페이지 대표 제목은 `h1` 하나
- 큰 섹션 제목은 `h2`
- 하위 제목은 `h3`, `h4`
- 제목의 시각적 크기는 CSS로 조정

HTML5의 아웃라인 알고리즘 관점에서는 `article`이나 `section` 내부에 별도의 `h1`을 둘 수 있다는 설명이 있지만, 실제 작업에서는 팀과 검색 엔진, 접근성 도구가 예측하기 쉬운 구조를 우선하는 편이 좋다.

## 7. Normal Flow와 Block

CSS의 기본 배치 흐름은 Normal Flow다. 일반적인 블록 요소는 위에서 아래로 쌓인다.

`p`, `div`, `h1` 같은 요소들은 한 줄 전체를 차지하는 상자처럼 배치된다. 새 블록 요소가 들어오면 기존 요소 아래로 밀려난다.

이 특성 때문에 어떤 요소의 크기가 변하면 아래 요소들의 위치를 다시 계산해야 한다. 특히 애니메이션으로 높이를 계속 바꾸면 매 프레임마다 아래 요소들을 다시 밀어야 하므로 성능 문제가 생길 수 있다.

실무 감각:

- 높이가 바뀌며 주변 레이아웃을 계속 밀어내는 애니메이션은 조심한다.
- 1초에 60번 계산해야 하는 상황이 생기면 사용자가 버벅임을 느낄 수 있다.
- 디자이너가 원하는 효과라도 브라우저 계산 비용을 고려해 대안을 제안할 수 있어야 한다.

## 8. Inline, Line Box, Anonymous Inline Box

블록 안의 글자는 그냥 떠 있는 것이 아니라, 브라우저 내부에서 라인 박스와 인라인 박스를 만든다.

개념적으로는 다음과 같다.

- 블록 박스: 한 줄 전체 폭을 차지하는 큰 상자
- 라인 박스: 텍스트 한 줄을 담당하는 상자
- 인라인 박스: 라인 안에서 글자 흐름을 구성하는 상자
- 익명 인라인 박스: 별도 태그 없이 텍스트만 있을 때 자동으로 생기는 인라인 박스

텍스트 중 일부만 선택해서 스타일을 주고 싶을 때 `div`를 쓰면 문단 흐름이 깨질 수 있다. `div`는 블록 성질을 가지기 때문이다.

```html
<p>이 문장 중 <span>중요한 부분</span>만 빨간색으로</p>
```

`span`은 의미를 강하게 부여하지 않고, 인라인 범위를 선택하기 위한 중립적인 태그다.

이 지점은 독학 과정에서 자주 대충 넘어가는 부분이다. `inline`을 단순히 "옆으로 붙는 요소"로만 외우면 baseline, line-height, 이미지 아래 여백 같은 문제를 이해하기 어렵다.

## 9. 인라인 태그를 고르는 기준

인라인 태그는 시각적 모양보다 의미를 기준으로 고르는 것이 좋다.

주요 기준은 다음과 같다.

- `span`: 의미 없이 특정 인라인 범위를 선택
- `strong`: 중요함, 강한 강조
- `em`: 문맥상 강조
- `b`: 중요도보다는 눈에 띄게 구분되는 텍스트
- `i`: 다른 분위기, 용어, 생각, 음성 등 구분되는 텍스트
- `q`: 짧은 인용
- `abbr`: 약어. `title` 속성으로 풀어쓴 의미 제공
- `time`: 시간, 날짜 정보
- `img`: 이미지. 인라인 성질을 가진다.
- `a`: 링크, 앵커

`b`와 `i`는 단순히 굵게 또는 기울임을 주기 위해 쓰는 태그가 아니다. 그런 표현은 CSS로 바꿀 수 있다.

핵심은 "어떤 모양을 만들 것인가"가 아니라 "이 텍스트가 문서 안에서 어떤 의미인가"를 기준으로 태그를 고르는 것이다.

## 10. 이미지 태그와 인라인 특성

`img`는 이미지를 불러오는 태그이며 인라인 성질을 가진다. 그래서 텍스트 흐름 안에 들어가면 글자와 함께 흐르고, baseline 기준으로 약간의 아래 여백이 생기는 것처럼 보일 수 있다.

```html
<img src="sample.png" alt="샘플 이미지" />
```

`alt`는 이미지가 보이지 않을 때의 대체 텍스트이자, 스크린 리더가 이미지를 설명할 때 사용하는 정보다. 장식용 이미지인지, 의미 있는 콘텐츠인지에 따라 `alt` 작성 방식도 달라져야 한다.

`img`, `meta`처럼 감쌀 대상이 없는 태그는 닫는 태그를 따로 두지 않는다. 이를 빈 요소 또는 self-closing 성격의 요소로 이해하면 된다.

실무에서는 `img`를 단순히 이미지를 보여주는 태그로만 보지 말고, 문서 내용에 포함되는 이미지인지, 장식용 표현인지 구분해야 한다. 의미 있는 이미지라면 HTML의 `img`가 자연스럽고, 장식용이라면 CSS background가 더 적합할 수 있다.

## 11. 속성과 이름 짓기

HTML 태그에는 속성을 붙일 수 있다.

```html
<body class="page_home"></body>
```

강의에서는 브라우저 개발자 도구에서 `class`가 JavaScript 객체의 `className` 같은 속성으로 보이는 것을 확인했다. HTML을 작성할 때는 `class`라고 적지만, JavaScript에서 다룰 때는 이름이 다르게 보일 수 있다.

이 부분은 HTML 문법만의 문제가 아니라, 같은 요소가 HTML, CSS, JavaScript, DOM API에서 서로 다른 관점으로 다뤄진다는 점을 보여준다.

강사 권장 스타일:

- HTML/CSS 쪽 이름은 소문자 중심으로 작성
- 단어 구분은 언더바 같은 구분자를 사용
- HTML 속성값은 큰따옴표 사용
- JavaScript에서는 camelCase와 작은따옴표를 쓰는 식으로 영역별 스타일을 구분

중요한 것은 특정 스타일이 절대 규칙이라는 뜻이 아니라, 팀이나 프로젝트의 규칙을 정하고 일관되게 지키는 것이다.

## 12. 태그 수와 성능

불필요한 태그를 과하게 감싸는 것은 성능에도 영향을 줄 수 있다.

예를 들어 상품 1만 개를 렌더링하는 리스트가 있다고 할 때, 상품 하나를 50개 태그로 만들 수 있는데 150개 태그로 만들면 브라우저가 처리해야 할 DOM 수가 크게 늘어난다.

서버 성능은 서버 증설로 해결할 수 있는 영역도 있지만, 브라우저가 사용자의 컴퓨터에서 처리해야 하는 DOM과 렌더링 비용은 프론트엔드 코드의 영향을 직접 받는다.

따라서 필요한 의미와 구조만 남기고, 불필요하게 한 번 더 감싸는 습관은 줄이는 것이 좋다.

다만 태그 수를 줄이는 것만 목표가 되어서는 안 된다. 의미 구조, 접근성, 스타일링의 안정성을 위해 필요한 태그는 남겨야 한다. 핵심은 적은 태그가 아니라 필요한 태그만 쓰는 것이다.

## 13. 1일차 실무 감각

1일차에서 반복된 실무 감각은 다음과 같다.

1. HTML은 화면 장식이 아니라 정보 구조를 만드는 언어다.
2. 태그는 브라우저가 DOM을 만들기 위한 단서다.
3. 브라우저는 HTML을 위에서 아래로 읽고, JavaScript는 렌더링 흐름을 막을 수 있다.
4. 시맨틱 태그는 SEO뿐 아니라 접근성, 유지보수, 자동화 도구에도 영향을 준다.
5. block과 inline의 차이는 레이아웃, 성능, baseline 문제로 이어진다.
6. 이름 짓기와 태그 수는 코드 품질과 렌더링 비용에 영향을 준다.

정리하면 1일차의 핵심은 다음 한 문장이다.

> 퍼블리싱은 화면을 그리는 일이 아니라, 브라우저가 해석할 수 있는 정보 구조를 설계하고 렌더링 비용까지 고려하는 일이다.
