# [05.11] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

3일차는 2일차에서 다룬 CSS 선택자, 상속, 박스 모델을 실제 계산 문제로 확장한 강의다.

핵심은 CSS를 속성 이름의 암기 목록으로 보는 것이 아니라, 브라우저가 상자를 만들고 크기와 위치를 계산하는 방식으로 이해하는 것이다.

특히 `width: 100%`, margin, padding, border, `overflow`, `rem`, `line-height`, web font, background image는 실무에서 자주 버그를 만드는 지점이다. 이 속성들은 독립적으로 외우기보다 어떤 영역을 기준으로 계산되는지 함께 봐야 한다.

강의의 주요 관점은 다음과 같다.

- 선택자와 가상 선택자의 의미를 구분한다.
- 상속되는 속성과 상속되지 않는 속성을 구분한다.
- `width: 100%`, margin, padding, border가 함께 있을 때 기준 영역을 먼저 본다.
- margin collapse와 BFC처럼 눈에 잘 보이지 않는 계산 맥락을 이해한다.
- 글자 크기, 줄 높이, 웹 폰트, 이미지는 디자인 품질뿐 아니라 유지보수와 성능에도 영향을 준다.

## 2. 선택자와 약속된 이름

CSS 선택자는 크게 두 종류로 볼 수 있다.

- 개발자가 직접 지어 붙인 이름: class, id, 태그 이름
- CSS 명세와 브라우저가 약속해 둔 이름: pseudo-class, pseudo-element

예를 들어 `category-item` 같은 class는 개발자가 직접 붙인 이름이다.

```html
<ul class="category">
  <li class="category-item">...</li>
</ul>
```

반면 `:first-child`, `:last-child`, `:hover`, `:active`, `:visited` 같은 이름은 브라우저와 CSS가 이미 약속해 둔 상태 이름이다.

```css
.category-item:last-child {
  margin-bottom: 0;
}

a:hover {
  color: red;
}
```

가상 선택자는 별도의 class를 추가하지 않아도 요소의 상태나 위치를 기준으로 스타일을 줄 수 있게 해준다.

실무에서는 "내가 붙인 이름"과 "브라우저가 제공하는 상태 이름"을 구분하는 것이 중요하다. 이 구분이 되면 React 같은 환경에서도 불필요한 상태 class나 조건부 로직을 줄일 수 있다.

## 3. `::before`, `::after`

`::before`와 `::after`는 CSS로 가상 요소를 만들어 붙이는 기능이다.

```css
.category::before {
  content: "";
  display: block;
}
```

핵심은 `content` 속성이 필요하다는 점이다. `::before`나 `::after`만 쓰는 것이 아니라, 무엇을 넣을지 `content`로 지정해야 한다.

이 기능은 실제 HTML 태그를 추가하지 않고 말풍선 꼬리, 구분 기호, 아이콘 장식, 배경용 보조 요소를 만들 때 자주 쓰인다.

다만 문서의 의미를 담는 내용은 HTML에 있어야 한다. `::before`, `::after`는 의미 구조를 대신하기보다 장식적인 표현을 보조하는 용도로 쓰는 것이 좋다.

## 4. CSS 상속

상속은 부모 요소의 일부 CSS 값이 자식 요소에게 전달되는 메커니즘이다.

주로 글자와 관련된 속성에서 상속이 일어난다.

- `color`
- `font-size`
- `font-family`
- `font-weight`
- `line-height`

반대로 상자 크기, 배치, 배경과 관련된 속성은 일반적으로 상속되지 않는다.

- `width`
- `height`
- `margin`
- `padding`
- `border`
- `background-color`
- `background-image`
- `position`

상속의 우선순위는 다음처럼 이해하면 된다.

1. 직접 지정한 값이 상속받은 값보다 우선한다.
2. 가까운 조상의 값이 먼 조상의 값보다 우선한다.
3. 아무 값도 지정하지 않으면 상속 가능한 속성은 부모 값을 따라간다.

글자 관련 속성이 상속되는 이유는 효율성 때문이다. 모든 `p`, `h1`, `span`에 글자색과 서체를 반복해서 지정하면 비효율적이다.

반대로 상자 크기나 배경색이 자동으로 자식에게 상속되면 레이아웃이 쉽게 무너질 수 있다. 그래서 배치와 박스 관련 속성은 대부분 상속되지 않는 편이 자연스럽다.

## 5. 박스 모델과 Width 계산

CSS 박스는 바깥에서 안쪽으로 `margin`, `border`, `padding`, `content` 영역을 가진다.

```text
margin
border
padding
content
```

block 요소는 기본적으로 부모의 사용 가능한 너비를 채우려 한다. 브라우저는 브라우저 창, `html`, `body`, 자식 상자의 순서로 너비를 계산하고, 각 단계에서 border, padding, margin을 반영한다.

개발자 도구에서 요소를 선택하면 content, padding, border, margin이 색상별로 구분된다. 이 표시를 보면서 실제 어떤 영역이 크기 계산에 참여하는지 확인해야 한다.

독학 단계에서는 `width`를 눈에 보이는 박스 전체 크기로 생각하기 쉽다. 하지만 기본값인 `content-box`에서는 `width`가 content 영역 기준이므로, padding과 border가 바깥으로 더해진다.

## 6. `width: 100%`, `box-sizing`, margin의 관계

`width: 100%`는 눈에 보이는 전체 박스 크기를 부모에 딱 맞춘다는 뜻이 아니다.

기본 `box-sizing: content-box` 상태에서는 `width`가 content 영역 기준이다.

```css
.box {
  width: 100%;
  padding: 10px;
  border: 4px solid black;
}
```

이 경우 `.box`의 content 영역이 부모 content 영역의 100%가 된다. 여기에 padding과 border가 추가로 붙기 때문에 실제 시각적 크기는 부모보다 커지고, 가로 스크롤이 생길 수 있다.

보통은 `border-box`를 사용해 이 문제를 줄인다.

```css
* {
  box-sizing: border-box;
}
```

`border-box`를 쓰면 `width`와 `height`가 content, padding, border를 포함한 전체 박스 기준으로 계산된다.

하지만 margin은 `border-box` 밖에 있다.

```css
.box {
  box-sizing: border-box;
  width: 100%;
  margin: 10px;
}
```

이 경우 padding과 border는 width 안에 들어오지만, margin은 바깥에 추가되므로 다시 가로 스크롤이 생길 수 있다.

필요한 간격이 자식의 바깥 여백인지, 부모의 안쪽 여백인지 판단해야 한다. 부모 안쪽 공간이 목적이라면 자식 margin보다 부모 padding이 더 안정적일 수 있다.

```css
body {
  padding: 10px;
}

.box {
  width: 100%;
  box-sizing: border-box;
}
```

핵심은 `width: 100%`나 `border-box`를 외우는 것이 아니라, 필요한 공간이 어느 상자의 어느 영역인지 판단하는 것이다.

## 7. Height 자동 계산과 Overflow

높이는 너비와 다르게 기본적으로 내부 콘텐츠를 감싸는 방향으로 계산된다.

예를 들어 한 줄의 `line-height`가 24px이고 두 줄이 있다면 content 높이는 대략 48px이 된다.

```css
p {
  line-height: 24px;
}
```

고정 높이를 지정하지 않으면 내용이 늘어날 때 상자도 함께 커진다. 반대로 `height: 100px`처럼 고정하면 내용이 더 많아졌을 때 콘텐츠가 상자 밖으로 넘칠 수 있다.

```css
.box {
  height: 100px;
  overflow: hidden;
}
```

`overflow`는 넘친 부분을 처리하는 속성이지만, 단순히 화면을 자르는 역할만 하지는 않는다. 뒤에서 다루는 BFC처럼 레이아웃 계산 맥락에도 영향을 줄 수 있으므로 이유를 알고 써야 한다.

## 8. Margin Collapse와 Auto

margin은 padding과 다르게 위아래 방향에서 겹칠 수 있다.

예를 들어 위 요소의 `margin-bottom`이 20px이고 아래 요소의 `margin-top`도 20px이라면, 둘 사이 간격은 40px이 아니라 20px이 될 수 있다. 이것을 margin collapse라고 한다.

```text
20px + 20px = 20px
30px + 20px = 30px
```

부모에게 border, padding 같은 경계가 없으면 자식의 수직 margin이 부모 밖으로 빠져나오는 것처럼 보일 수도 있다. 이것도 margin collapse의 한 종류로 이해할 수 있다.

해결 방법으로는 부모에게 padding이나 border를 주거나, 필요할 때 `overflow: hidden`으로 새로운 계산 맥락을 만드는 방법이 있다. 다만 `overflow: hidden`은 BFC를 만들기 때문에 단순한 눈속임으로만 쓰면 안 된다.

margin과 padding의 차이는 다음처럼 정리할 수 있다.

- padding은 상자 안쪽 여백이며, 자기 상자의 내부 공간에 포함된다.
- margin은 상자 바깥 여백이며, 다른 상자와의 거리다.

normal flow에서 좌우 margin을 `auto`로 주면 block 요소를 가로 가운데 정렬할 수 있다.

```css
.container {
  max-width: 1200px;
  margin: 0 auto;
}
```

이때 중요한 것은 위아래 auto가 아니라 좌우 auto다. 좌우 auto는 부모의 남는 공간을 계산해 양쪽에 나눠 넣는 방식이므로, 보통 명시적인 `width` 또는 `max-width`와 함께 쓴다.

## 9. BFC와 렌더링 흐름

BFC는 Block Formatting Context의 약자다. block 요소들이 어떤 규칙으로 자리 잡고 크기를 계산할지 정하는 독립적인 계산 맥락으로 볼 수 있다.

브라우저 렌더링 흐름은 대략 다음과 같다.

```text
HTML 파싱
-> DOM 생성
-> CSSOM 생성
-> Render Tree 생성
-> Layout 계산
-> Paint
```

`display: none`인 요소는 DOM에는 있지만 render tree에는 올라오지 않는다. render tree에 없으면 layout과 paint 대상도 아니다.

그래서 `display: none`은 애니메이션이 어렵다. 보였다가 사라지는 fade 효과를 만들려면 `opacity`, `visibility`, `transition` 같은 속성을 활용하는 편이 자연스럽다.

```css
.fade {
  opacity: 0;
  visibility: hidden;
  transition: opacity 0.3s ease;
}
```

`overflow: hidden`은 넘치는 콘텐츠를 자르는 속성이지만, 동시에 BFC를 만들어 margin collapse를 막는 효과도 낸다.

문제는 이 계산이 공짜가 아니라는 점이다. 많은 상자에 무심코 `overflow: hidden`을 넣으면 렌더링 비용이 늘 수 있다. 실무에서는 "넘친 내용을 자르려는 것인지", "margin collapse를 막으려는 것인지", "다른 해결책이 더 안전한지"를 구분해야 한다.

## 10. `rem`, `line-height`, 타이포그래피 토큰

`rem`은 root element, 즉 `html`의 font-size를 기준으로 하는 단위다.

```css
html {
  font-size: 18px;
}

h4 {
  font-size: 2rem;
}
```

위 예시에서 `h4`의 글자 크기는 36px이 된다.

```text
18px * 2 = 36px
```

`rem`을 쓰면 전체 글자 크기 체계를 한 번에 조정하기 쉽다. 예를 들어 기준이 18px이고 원하는 크기가 24px이라면 `24 / 18 = 1.333333rem`처럼 환산할 수 있다.

`line-height`는 px로 지정할 수도 있고, 단위 없는 숫자로 지정할 수도 있다.

```css
body {
  font-size: 18px;
  line-height: 1.4;
}
```

단위 없는 `line-height`는 각 요소의 font-size에 곱해져 계산된다. `font-size: 18px`이면 25.2px, `font-size: 24px`이면 33.6px이 된다.

단위 없는 `line-height`를 상속시키면 글자 크기가 다른 요소들도 같은 비율의 줄 높이를 유지할 수 있다.

실무 디자인 시스템에서는 글자 크기를 무작위로 쓰지 않는다. `Title 1`, `Title 2`, `Headline`, `Body`, `Callout`, `Subhead`, `Footnote`처럼 이름이 붙은 타입 스타일을 만들고, 각 스타일에 font-size, line-height, weight를 정한다.

중요한 것은 화면마다 임의로 `27px`, `29px` 같은 값을 쓰는 것이 아니라, 미리 정한 몇 개의 값을 재사용하는 것이다.

## 11. Web Font

브라우저가 특정 서체로 글자를 그리려면 사용자의 기기에 그 서체가 있어야 한다. 없으면 fallback font가 사용된다.

```css
body {
  font-family: "Noto Sans KR", sans-serif;
}
```

원하는 서체를 사용자에게 내려주기 위해 web font를 사용한다. 대표적으로 Google Fonts 같은 서비스를 이용할 수 있다.

```css
@import url("https://fonts.googleapis.com/css2?family=Noto+Sans+KR&display=swap");
```

주의할 점은 한글 웹 폰트는 용량이 크다는 것이다. 여러 서체와 여러 굵기를 무분별하게 쓰면 페이지 로딩이 무거워진다.

웹 폰트는 시각 품질을 얻는 대신 다운로드 비용을 추가하는 선택이다. 실무에서는 필요한 서체와 weight만 신중하게 사용하고, 로딩 성능과 fallback 상태까지 함께 봐야 한다.

## 12. Background 속성

모든 상자는 기본적으로 배경색이 투명하다. 부모 상자에 배경색을 주면 자식의 배경이 투명한 부분을 통해 부모 배경이 보인다.

배경 이미지는 상자의 바닥에 이미지를 채워 넣는 개념이다.

```css
.box {
  background-color: transparent;
  background-image: url("./images/radio.png");
  background-repeat: no-repeat;
  background-position: right bottom;
}
```

경로에서 `./`는 현재 파일 위치를 기준으로 시작한다는 뜻이다.

배경은 아래에서 위로 쌓인다.

```text
background-color
-> background-image
-> 자식 콘텐츠
```

배경 이미지는 기본적으로 반복된다. 패턴 이미지는 반복이 필요할 수 있지만, 아이콘이나 장식 이미지는 반복을 꺼야 하는 경우가 많다.

위치는 `background-position`으로 조정한다. 픽셀 값을 쓸 수도 있지만, 반응형에서는 `right`, `bottom`, `center` 같은 키워드가 더 안정적인 경우가 많다.

## 13. Background Image와 Image Tag

배경 이미지는 CSS 표현이다. 상자의 장식이나 시각적 분위기를 만들 때 적합하다.

```css
.hero {
  background-image: url("./images/visual.jpg");
}
```

반면 `img` 태그는 문서 내용 자체로서의 이미지에 적합하다.

```html
<img src="./images/photo.jpg" alt="설명" />
```

의미 있는 콘텐츠라면 `img` 태그를 쓰고, 장식용이라면 background image를 쓰는 쪽이 자연스럽다.

`img` 태그를 특정 박스 안에 꽉 차게 넣고 비율을 유지한 채 잘라내고 싶을 때는 `object-fit: cover`를 쓴다.

```css
.visual img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```

이는 background image에서 `background-size: cover`를 쓰는 것과 비슷한 효과를 `img` 태그에 적용하는 방식이다.

이미지를 어떤 방식으로 넣을지는 의미와 관리 기준으로 판단한다. 문서의 내용이면 `img`, 장식이면 background image, 요청 수를 줄이기 위한 작은 아이콘 묶음이면 스프라이트 같은 방식을 고려할 수 있다.

다만 아이콘이나 로고를 하나의 파일로 묶으면 서버 요청 수를 줄일 수 있는 대신, 개별 이미지만 수정하거나 교체하기 어렵고 CSS로 좌표와 크기를 맞춰야 하므로 관리 비용이 늘어난다. 성능을 위해 파일 제어와 유지보수 편의성을 일부 희생하는 선택이다.

## 14. 3일차 실무 감각

3일차에서 반복된 실무 감각은 다음과 같다.

1. CSS를 외우지 말고 브라우저가 계산하는 방식을 이해한다.
2. `width: 100%`, margin, padding, border가 함께 있을 때는 어떤 영역이 기준인지 먼저 본다.
3. margin은 padding과 다르며, 위아래 방향에서 collapse와 parent escape 현상이 생길 수 있다.
4. `overflow: hidden`은 문제 해결에 도움이 되지만 BFC와 성능 영향을 함께 생각해야 한다.
5. 글자 크기와 줄 높이는 디자인 시스템의 토큰으로 관리하는 것이 좋다.
6. 웹 폰트는 시각 품질을 높이지만 용량과 로딩 비용을 관리해야 한다.
7. 이미지는 의미 있는 콘텐츠인지 장식인지에 따라 `img`와 background image를 구분한다.
8. 성능을 위한 선택은 유지보수 비용이나 관리 복잡도를 함께 만든다.

정리하면 3일차의 핵심은 다음 한 문장이다.

> 퍼블리싱은 명령어 암기가 아니라, 브라우저가 상자를 어떻게 계산하고 그리는지 이해하는 일이다.
