# [05.12] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

4일차는 이미지 처리, flex 배치, 디자인 시스템, CSS 변수, 컴포넌트식 퍼블리싱을 실제 작업 흐름 안에서 연결한 강의다.

핵심은 디자인을 단순히 눈에 보이는 화면으로 따라 치는 것이 아니라, 재료와 부품, 배치 규칙, 상태 변형을 가진 시스템으로 읽는 것이다.

강의의 주요 관점은 다음과 같다.

- 이미지 형식과 파일 구성은 서버 요청 수와 렌더링 비용에 영향을 준다.
- flex는 모든 레이아웃에 무조건 쓰는 도구가 아니라, 필요한 배치 문제를 해결할 때 선택하는 도구다.
- Figma의 스타일, 변수, 컴포넌트는 CSS 변수와 class 구조로 옮길 수 있다.
- 디자인 토큰은 색상이나 수치를 직접 박아 넣는 대신 의미 있는 이름으로 관리하는 방식이다.
- 좋은 퍼블리싱은 시각적 구현뿐 아니라 구조화, 재사용성, 유지보수 비용까지 고려한다.

## 2. 이미지 형식과 서버 요청 수

강의 초반에는 네이버 같은 실제 서비스의 네트워크 요청을 보면서, 작은 이미지들이 페이지 성능에 어떤 영향을 주는지 설명했다.

브라우저는 필요한 파일을 하나씩 서버에 요청한다. 아이콘이나 작은 이미지가 각각 별도 파일로 쪼개져 있으면 요청 수가 늘고, 네트워크 탭에는 그 요청 목록이 그대로 보인다.

작은 아이콘이나 로고를 하나의 파일로 묶어 사용하는 이유는 이 요청 수를 줄이기 위해서다. 예를 들어 이미 270개 요청이 있는 페이지에 작은 이미지 100개가 추가되면 요청 부담이 크게 늘 수 있다.

다만 이 방식은 트레이드오프가 있다.

- 얻는 것: 서버 요청 수 감소, 초기 로딩 비용 절감 가능
- 잃는 것: 개별 아이콘만 수정하기 어려움, CSS 좌표 관리 필요, 일부 아이콘만 필요한 화면에서도 묶인 파일 전체를 내려받을 수 있음

즉, 작은 이미지를 묶는 방식은 성능을 위해 파일 제어와 유지보수 편의성을 일부 희생하는 선택이다.

## 3. SVG와 Bitmap을 고르는 기준

SVG는 확대해도 깨지지 않고, 단순한 아이콘이나 로고에는 적합하다. 하지만 모든 이미지를 SVG로 처리하는 것이 항상 좋은 것은 아니다.

그라데이션, 번짐, 복잡한 장식, 많은 요소를 SVG로 그리면 브라우저가 그 벡터를 계산하고 렌더링해야 한다. 이 경우 오히려 렌더링 비용이 커질 수 있다.

네이버 같은 서비스가 복잡한 이미지를 SVG로 처리하지 않는 것은 SVG를 몰라서가 아니라, 그 상황에서 더 나은 선택지가 있기 때문이다.

실무 판단은 다음처럼 정리할 수 있다.

- 단순 아이콘, 로고, 선명한 벡터 그래픽: SVG가 적합
- 복잡한 질감, 사진, 그라데이션이 많은 이미지: PNG, JPG, WebP 같은 bitmap 계열이 적합할 수 있음
- 작은 아이콘 묶음: 요청 수와 관리 비용을 비교해 sprite나 개별 파일을 선택

이미지 형식은 "어떤 게 최신인가"보다 "그 이미지가 어떻게 쓰이고, 브라우저가 어떤 비용을 부담하는가"로 판단해야 한다.

## 4. Flex는 필요할 때 선택하는 레이아웃 모드다

강의에서는 flex를 가로/세로 배치와 정렬 문제를 해결하는 도구로 다뤘다.

flex에서는 같은 방향과 반대 방향을 구분해야 한다. `flex-direction`이 `row`이면 주축은 가로, 교차축은 세로가 된다. `column`이면 이 관계가 바뀐다.

대표적인 속성은 다음과 같다.

- `flex-direction`: 주축 방향 결정
- `justify-content`: 주축 방향 정렬
- `align-items`: 교차축 방향 정렬

```css
.box {
  display: flex;
  justify-content: space-between;
  align-items: center;
}
```

`space-between`은 남는 공간을 아이템 사이에 균등하게 배분한다. 아이템 개수와 크기를 바탕으로 브라우저가 간격을 계산하기 때문에, 고정 margin을 일일이 넣지 않아도 되는 경우가 있다.

하지만 flex를 무조건 쓰면 안 된다. 위에서 아래로 자연스럽게 쌓이는 normal flow로 충분한 경우에는 flex를 쓰지 않는 편이 더 단순하다.

실무에서는 "가운데 정렬해야 하니까 flex"가 아니라, 현재 필요한 배치가 normal flow로 가능한지, flex가 실제로 해결해야 할 문제가 있는지를 먼저 판단해야 한다.

## 5. Flex 안에서의 `auto`

normal flow에서 block 요소의 `width: auto`는 보통 부모의 사용 가능한 너비를 채우는 방향으로 계산된다.

하지만 flex container 안에서는 상황이 달라진다. flex item의 `auto`는 자기 내용만큼 줄어든 상태로 계산될 수 있다.

강의에서는 이 차이를 다음처럼 설명했다.

- normal flow의 block: 기본적으로 가로를 꽉 채우며 아래로 쌓임
- flex item: 내용 크기와 flex 규칙에 따라 줄어들거나 늘어남

이 차이를 모르면 같은 `auto`인데 왜 어떤 곳에서는 꽉 차고, 어떤 곳에서는 내용만큼만 줄어드는지 이해하기 어렵다.

flex를 쓸 때는 "무슨 방향으로 배치할 것인가"뿐 아니라, flex item의 기본 크기가 어떻게 계산되는지도 함께 봐야 한다.

## 6. 디자인을 코드로 옮기기 전에 구조를 그린다

강의에서는 실습 전에 상자를 어떻게 구성할지 먼저 그려보라고 했다.

바로 CSS를 작성하면 눈에 보이는 위치를 맞추는 데만 집중하기 쉽다. 그러면 부모-자식 관계, 반복 부품, 배치 책임이 뒤섞여 나중에 수정 비용이 커진다.

먼저 해야 할 일은 다음과 같다.

1. 화면을 큰 상자 단위로 나눈다.
2. 각 상자의 이름과 역할을 정한다.
3. 부모-자식 관계를 그린다.
4. normal flow로 가능한지, flex가 필요한지 판단한다.
5. 반복될 수 있는 구조는 컴포넌트 후보로 본다.

이날의 핵심은 구조를 먼저 정한 뒤 CSS를 붙이는 순서다. 이 감각은 HTML/CSS뿐 아니라 React 컴포넌트를 나눌 때도 이어진다.

## 7. 타이포그래피는 수치를 그대로 박지 않는다

디자인 검수에서는 글자 크기, 굵기, 줄 높이 차이가 바로 드러난다. 그래서 Figma의 값을 보고 font-size, font-weight, line-height를 맞추는 작업이 필요하다.

하지만 모든 `line-height`를 px 단위로 각각 박아 넣으면 값은 맞아 보여도 규칙이 흩어진다. 비슷한 비율이 반복된다면 단위 없는 `line-height`를 공통 규칙으로 두고, 각 요소가 자기 `font-size`에 맞춰 계산되게 하는 편이 관리하기 쉽다.

핵심은 디자이너가 준 숫자를 기계적으로 전부 복사하는 것이 아니라, 그 안에 있는 규칙을 읽어 코드 구조로 옮기는 것이다.

## 8. Figma와 브라우저의 차이는 인정해야 한다

Figma에서 보이는 타이포그래피와 브라우저에서 렌더링되는 결과가 항상 1px 단위로 같지는 않다.

서체 자체의 메트릭, Figma의 렌더링 방식, 브라우저별 폰트 렌더링 차이 때문에 완전히 같은 결과를 보장하기 어렵다.

강의에서는 이런 차이가 있을 때 무조건 맞추려고 하기보다, 어디까지가 구현 가능한 차이인지 설명할 수 있어야 한다고 했다.

특히 Figma의 vertical trim 같은 기능은 브라우저 CSS에서 안정적으로 대응되지 않을 수 있다. 디자인 파일에서 해당 옵션이 켜져 있으면 웹 구현에서 계속 어긋날 수 있으므로, 초기에 디자이너와 조율하는 것이 좋다.

실무에서는 디자인 수치를 최대한 맞추되, 브라우저와 Figma의 렌더링 차이를 설명 가능한 한계로 봐야 한다. "1px도 무조건 맞춘다"보다 "어떤 차이가 왜 생기는지 설명하고 조율한다"가 더 안전한 태도다.

## 9. 링크와 블록 콘텐츠

강의에서는 카드 전체가 클릭되어야 하는 상황에서 `a` 태그 안에 여러 요소를 넣는 구조를 다뤘다.

과거에는 inline 요소 안에 block 요소를 넣는 것이 제한적이었지만, HTML5에서는 `a`가 transparent content model을 가지므로 여러 블록 콘텐츠를 감쌀 수 있다.

카드 전체가 링크라면 다음처럼 구조화할 수 있다.

```html
<a class="music-card" href="#">
  <img src="..." alt="" />
  <div class="music-text">...</div>
</a>
```

다만 `a` 태그는 기본적으로 파란색과 밑줄 스타일을 가진다. 이 기본 스타일이 내부 텍스트로 상속되므로, 카드형 링크에서는 링크 자체에 색상과 밑줄 초기화를 주는 편이 안정적이다.

```css
.music-card {
  color: inherit;
  text-decoration: none;
}
```

기본 스타일을 지우는 것은 편하지만, 링크임을 알아볼 수 있는 다른 시각적 단서나 hover/focus 상태도 함께 설계해야 한다.

## 10. 실습에서 드러난 박스 계산 문제

실습 중 디자인에서 측정한 너비를 그대로 `width`에 넣었는데도 화면이 더 넓어지는 문제가 있었다.

원인은 padding과 border가 width 바깥으로 더해지는 `content-box` 계산 방식이었다.

`width: 1092px`에 좌우 `padding: 25px`을 더하면 실제 시각적 너비는 더 커질 수 있다. 이 상황에서는 `box-sizing: border-box`로 padding과 border를 지정한 크기 안에 포함시키는 편이 맞다.

이 사례의 핵심은 브라우저가 어떤 영역을 기준으로 크기를 계산하는지 확인하는 것이다. 디자인 수치를 그대로 넣어도 기준 영역을 잘못 잡으면 화면은 맞지 않는다.

디버깅용 border를 넣을 때도 주의해야 한다. border는 실제 박스 크기에 영향을 줄 수 있으므로, 확인용 선 때문에 레이아웃이 달라질 수 있다.

## 11. CSS는 마크업 구조 순서대로 작성한다

강의에서는 CSS를 작성할 때 HTML 구조 순서를 따라가는 습관을 권했다.

HTML에서 부모와 자식, 형제 관계가 잡혀 있다면 CSS도 그 흐름을 따라 작성하는 편이 읽기 쉽다.

이 방식은 머릿속에 DOM 구조를 유지하게 해주고, 어떤 스타일이 어느 요소에 대응되는지 찾기 쉽게 만든다. 반대로 CSS를 화면에 보이는 순서나 생각나는 순서대로 흩어놓으면, 나중에 수정할 때 구조를 다시 추적해야 한다.

퍼블리싱을 잘한다는 것은 단순히 화면을 맞추는 것이 아니라, 구조가 보이는 코드를 만드는 것이다.

## 12. 디자인 시스템은 재료에서 페이지로 확장된다

강의 후반부는 Figma 디자인 시스템을 코드로 옮기는 흐름으로 이어졌다.

디자인 시스템은 대략 다음 순서로 구성된다.

```text
원재료
-> 재료
-> 부품
-> 페이지
```

원재료는 색상, 글꼴, 숫자, radius 같은 가장 작은 값이다. 재료는 아이콘, 그림자, 타이포그래피 스타일처럼 조금 더 구체화된 단위다. 부품은 버튼, 타이틀, 카드 같은 컴포넌트이고, 페이지는 이 부품들을 조립한 결과다.

코드에서는 원재료가 CSS 변수나 design token이 되고, 재료는 typography class나 shadow, icon 규칙이 된다. 부품은 button, app-title, card 같은 컴포넌트가 되며, 페이지는 이 부품을 조립한 결과다.

디자인을 이 구조로 읽으면, 무엇을 변수로 빼야 하고 무엇을 class나 component로 만들어야 하는지 판단하기 쉬워진다.

## 13. 디자인 토큰은 의미 있는 이름이어야 한다

디자인 토큰은 색상이나 숫자에 의미 있는 이름을 붙여 재사용하는 방식이다.

강의에서는 색상 이름을 단순히 `blue1`, `blue2`처럼 짓는 것이 위험하다고 설명했다. 색이 바뀌면 이름과 실제 의미가 어긋나기 때문이다.

예를 들어 `brand-1`, `primary`, `outline`, `card`, `radius`처럼 역할을 기준으로 이름을 붙이면 값이 바뀌어도 토큰의 의미는 유지된다. Figma에서는 Variables로 이런 값을 관리할 수 있고, 코드에서는 CSS 변수로 옮길 수 있다.

```css
:root {
  --app-bg: #f7f7f7;
  --primary: #3182f6;
  --outline: rgba(0, 0, 0, 0.1);
  --radius-card: 16px;
}
```

CSS 변수는 `:root`에 두면 문서 전역에서 접근하기 쉽다.

토큰 이름은 디자이너와 개발자가 함께 맞추는 것이 좋다. 디자인 파일에서는 `primary`인데 코드에서는 `main-blue`라고 부르면 소통 비용이 늘어난다.

## 14. 라이트 모드와 다크 모드를 위한 변수화

디자인 시스템에서 변수화가 중요한 이유 중 하나는 라이트 모드와 다크 모드를 쉽게 전환하기 위해서다.

같은 `--app-bg` 토큰이라도 라이트 모드에서는 밝은 색, 다크 모드에서는 어두운 색을 가리키게 만들 수 있다.

```css
:root {
  --app-bg: #ffffff;
  --text-main: #111111;
}

[data-theme="dark"] {
  --app-bg: #111111;
  --text-main: #ffffff;
}
```

이렇게 하면 컴포넌트 코드는 `var(--app-bg)`만 사용하고, 실제 색상 값은 테마에 따라 바뀐다.

중요한 것은 특정 회색값을 화면마다 직접 쓰는 것이 아니라, 역할을 가진 이름을 통해 색을 관리하는 것이다. 그래야 다크 모드나 브랜드 색 변경이 생겨도 수정 범위가 줄어든다.

## 15. CSS 파일 분리와 연결

지금까지는 HTML 내부에 CSS를 함께 작성했지만, 실제 프로젝트에서는 CSS 파일을 분리하는 편이 관리하기 좋다.
CSS 파일에는 문자 인코딩을 명시할 수 있다.

```css
@charset "UTF-8";
```

대부분의 경우 없어도 동작하지만, CSS 안에 한글이나 `content` 문자열이 들어갈 수 있으므로 안전하게 명시할 수 있다.

CSS 파일을 분리하면 HTML은 구조에 집중하고, CSS는 표현 규칙에 집중할 수 있다.

## 16. 기본 타이포그래피는 상속으로 내린다

디자인 시스템을 코드로 옮길 때, 전체 기본 글꼴과 글자 크기, 줄 높이는 상위 요소에 두고 상속시키는 방식이 좋다.

```css
html {
  font-family: "Inter", sans-serif;
  font-size: 14px;
  font-weight: 500;
  line-height: 1.2;
}
```

글밥이 가장 많은 본문 스타일을 기준으로 잡고, 필요한 컴포넌트만 별도 class로 변형한다.

메인 타이틀이나 서브 타이틀처럼 예외가 필요한 부품만 별도 class로 변형한다. `line-height`가 이미 공통 규칙으로 맞는다면 각 컴포넌트마다 다시 지정하지 않아도 된다.

중요한 것은 Figma 값 하나하나를 기계적으로 복사하는 것이 아니라, 공통 규칙과 예외를 구분하는 것이다.

## 17. 컴포넌트 이름은 디자인 파일과 맞춘다

강의에서는 Figma에 있는 `main-title`, `sub-title`, `app-title` 같은 부품 이름을 코드에서도 그대로 사용하는 방식을 설명했다.

디자이너는 Figma에서 부품을 이름으로 고르고, 개발자는 그 이름을 class나 component 이름으로 옮긴다. 이름이 맞으면 소통 비용이 줄어든다.

예를 들어 디자이너는 `sub-title`이라고 부르는데 개발자가 `small-title`, `low-title`처럼 다른 이름을 쓰면, 같은 대상을 두고 서로 다른 언어를 쓰게 된다.

실무에서는 이름이 곧 협업 계약이다. 디자인 파일의 부품 이름, CSS class 이름, React component 이름, 문서와 작업 티켓의 용어가 최대한 같은 방향을 가리켜야 한다.

## 18. 디자인 도구의 Auto Layout과 코드의 Normal Flow

Figma의 Auto layout은 코드의 normal flow나 flex와 유사한 사고방식을 가진다.

예를 들어 `a-day` 같은 하루 단위 부품을 복제하면 아래로 자연스럽게 쌓이고, 그 안에 제목과 목록이 들어간다. 이것은 HTML에서 부모 안에 자식이 들어가고, normal flow에 따라 아래로 쌓이는 구조와 비슷하다.

디자인 파일이 이 구조를 잘 반영하고 있으면 개발자는 그 의도를 코드로 옮기기 쉽다.

하지만 실제 실무에서는 디자인 파일이 항상 이렇게 정리되어 있지 않을 수 있다. 변수 처리도 안 되어 있고, 부품 처리도 안 되어 있고, 오토레이아웃도 불완전할 수 있다.

이때 개발자는 코드에서라도 토큰, 반복 구조, 부모-자식 관계를 다시 잡아야 한다. 디자인의 의도가 보이지 않으면 먼저 의도를 추론하고 확인하는 과정도 필요하다.

디자인 시스템이 잘 갖춰질수록 AI나 MCP 기반 코드 생성 품질도 높아진다. 반대로 디자인 구조가 엉켜 있으면 자동 생성 결과도 엉키기 쉽다.

## 19. 4일차 실무 감각

4일차에서 반복된 실무 감각은 다음과 같다.

1. 이미지 형식과 파일 구성은 요청 수와 렌더링 비용을 함께 보고 결정한다.
2. flex는 무조건 쓰는 것이 아니라, normal flow로 부족할 때 선택한다.
3. Figma의 수치를 그대로 복사하지 말고, 그 안의 규칙을 읽어 CSS 구조로 옮긴다.
4. line-height, font-size, font-weight는 공통 규칙과 예외를 나눠 관리한다.
5. 디자인 토큰은 색상값이 아니라 역할과 의미를 이름으로 관리하는 방식이다.
6. CSS 변수는 디자인 토큰을 코드에 옮기는 기본 도구가 된다.
7. 컴포넌트 이름은 디자인 파일과 코드가 같은 언어를 쓰도록 맞추는 것이 좋다.
8. 좋은 퍼블리싱은 화면을 맞추는 것뿐 아니라, 구조화와 재사용성을 설계하는 일이다.

정리하면 4일차의 핵심은 다음 한 문장이다.

> 디자인을 코드로 옮긴다는 것은 화면을 베끼는 일이 아니라, 디자인 시스템의 재료와 부품, 배치 규칙을 HTML/CSS 구조로 번역하는 일이다.
