# [05.20] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

Normal flow, flex, box model 이해를 바탕으로 `grid`, `position`, responsive web design의 선택 기준을 정리한다.

배치 모드는 상황에 맞게 골라야 한다. 모든 배치를 flex로 억지로 만들 수는 있지만, 격자 구조는 grid가 더 자연스럽고, 흐름에서 빠져 떠야 하는 요소는 position이 필요하다. 화면 크기에 따라 구조 자체가 바뀌는 순간에는 media query가 필요하다.

반드시 잡아야 할 판단 기준은 다음과 같다.

- `display`는 요소가 어떤 렌더링/배치 세계에 참여할지 바꾼다.
- Grid는 행과 열이 있는 반복 구조에 적합하다.
- Position은 normal flow, flex, grid의 흐름에서 빠져 별도 기준 사각형을 기준으로 움직인다.
- `relative`, `absolute`, `fixed`의 차이는 기준 사각형이 무엇인지에 있다.
- `top`, `right`, `bottom`, `left`는 좌표가 아니라 기준 사각형의 네 면과 자신의 네 면 사이의 관계다.
- 반응형 웹은 화면 폭이 늘어나는 문제가 아니라, 특정 지점에서 레이아웃 규칙을 바꾸는 문제다.

핵심은 속성을 외우는 것이 아니라 normal flow, flex, grid, position, media query가 각각 어떤 문제를 해결하는지 구분하는 것이다.

## 2. `display`는 배치 세계를 바꾼다

`display`는 단순히 요소를 보이게 하거나 숨기는 속성이 아니라, 요소가 어떤 레이아웃 계산 방식에 참여할지를 정한다.

`display: block`, `display: inline`, `display: flex`, `display: grid`, `display: none`은 같은 속성의 서로 다른 값이다. 값이 바뀌면 요소의 박스가 normal flow에 참여할지, flex formatting context를 만들지, grid formatting context를 만들지, 렌더 tree에서 빠질지가 달라진다.

특히 `display: none`은 화면에서만 안 보이는 정도가 아니라, 렌더 tree에서 빠지는 쪽에 가깝다. DOM에는 정보가 남아 있어도 브라우저가 실제로 그릴 대상에서는 제외된다.

반대로 `display: flex`나 `display: grid`는 부모 요소 안에 새로운 배치 계산 세계를 만든다. 자식 요소는 기존 normal flow가 아니라 flex나 grid의 규칙에 따라 다시 크기와 위치가 계산된다.

실무에서는 `display`를 "이 요소가 어떤 세계에 들어가는가"로 봐야 한다. 같은 HTML 구조라도 `display` 값이 바뀌면 브라우저가 계산하는 배치 방식이 달라진다.

## 3. Grid는 격자 구조일 때 선택한다

Grid는 행과 열이 있는 격자 구조를 만들기 위한 배치 도구다.

작은 가로/세로 정렬은 flex로 충분한 경우가 많다. 버튼과 텍스트를 나란히 놓거나, 카드 내부의 아이콘과 숫자를 맞추는 경우가 여기에 가깝다.

반면 캘린더, 상품 목록, 썸네일 목록처럼 일정한 칸이 반복되고, 항목 수가 늘거나 줄어도 그 칸에 맞춰 배치되어야 한다면 grid가 더 적합하다.

중요한 것은 "flex로도 가능하다"와 "flex가 좋은 선택이다"를 구분하는 것이다. 격자 구조를 flex로 억지로 만들 수는 있지만, grid가 제공하는 행/열 기준과 자동 배치의 이점을 버리게 된다.

Grid는 다음 상황에서 먼저 검토한다.

- 일정한 행과 열이 필요한 경우
- 항목 수가 바뀌어도 칸에 맞춰 자동 배치되어야 하는 경우
- 특정 아이템을 여러 칸에 걸치게 해야 하는 경우
- 목록 부모가 전체 배치 규칙을 책임지는 편이 자연스러운 경우

배치 도구를 고르는 기준은 속성 취향이 아니라 UI의 구조다.

## 4. Grid는 부모가 행과 열을 정의한다

Flex와 마찬가지로 grid도 부모에게 선언한다.

```css
.grid {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  grid-template-rows: repeat(3, 1fr);
}
```

부모가 grid container가 되면 그 안의 자식은 grid item으로 배치된다. 간격, 열 수, 행 높이, 자동 배치 규칙은 보통 개별 아이템이 아니라 목록 부모가 책임진다.

`grid-template-columns`는 가로 방향의 칸을, `grid-template-rows`는 세로 방향의 칸을 만든다. `fr`은 남는 공간을 나눠 갖는 grid 전용 단위로 볼 수 있고, `repeat(4, 1fr)`은 `1fr` 칸을 네 번 반복한다는 뜻이다.

이때 중요한 것은 수치를 많이 외우는 것이 아니라, 부모 grid가 몇 개의 열과 행을 만들고 그 공간을 어떤 비율로 나눌지 정하는 것이다.

## 5. Grid item은 자동 배치와 선 번호 배치를 모두 가진다

Grid container에 자식 요소가 여러 개 들어오면 브라우저는 지정된 격자 구조에 따라 아이템을 순서대로 채운다. 상품이 오늘은 33개이고 내일은 29개일 수 있는 목록에서도 이 자동 배치가 유용하다.

아이템 사이의 간격은 부모 grid에 `gap`을 주는 방식이 자연스럽다.

```css
.grid {
  display: grid;
  gap: 20px;
}
```

필요하면 특정 아이템을 선 번호 기준으로 배치할 수도 있다.

```css
.item-a {
  grid-column: 3 / 5;
  grid-row: 1 / 3;
}
```

이 코드는 `.item-a`가 column 3번 선부터 5번 선까지, row 1번 선부터 3번 선까지 차지한다는 뜻이다. 그래서 한 칸이 아니라 여러 칸을 가로지르는 배치도 가능하다.

다만 모든 grid 레이아웃을 선 번호로 직접 배치할 필요는 없다. 일반적인 목록은 자동 배치와 `gap`만으로 충분한 경우가 많고, 선 번호 배치는 특정 아이템만 특별한 위치나 크기를 가져야 할 때 쓰는 정밀 도구에 가깝다.

## 6. Grid도 항상 정답은 아니다

Grid는 격자 틀을 잡고 반복 항목을 배치하는 데 강하다. 브라우저가 행과 열의 규칙을 알고 있으므로, 반복 구조를 명확히 계산할 수 있다.

하지만 모든 레이아웃을 grid로 만들 필요는 없다. 작은 내부 정렬은 flex가 더 단순하고, 문서가 자연스럽게 위에서 아래로 흐르면 normal flow가 더 낫다. 화면 폭에 따라 몇 열이 필요한지, 어느 지점에서 구조가 바뀌는지도 함께 봐야 한다.

실무 기준은 다음처럼 정리할 수 있다.

- 문서가 자연스럽게 흐르면 normal flow를 유지한다.
- 한 방향 정렬이면 flex를 먼저 본다.
- 명확한 행/열 구조라면 grid를 본다.
- 흐름에서 빠져 떠야 하는 요소는 position을 검토한다.

Grid는 강력하지만, UI 구조가 격자를 요구할 때 선택해야 한다.

## 7. Position은 흐름에서 빠져나오는 배치 층이다

`position`은 normal flow, flex, grid의 배치 규칙에서 벗어나 요소를 별도의 위치 기준으로 띄우는 방식이다.

normal flow에서는 요소가 위에서 아래로 쌓이고, flex에서는 부모의 flex 규칙을 따르며, grid에서는 격자 규칙을 따른다. 그런데 배지, 말풍선, 고정 메뉴, 모달, 알림창처럼 이 흐름의 강제에서 벗어나야 하는 요소가 있다.

이때 position을 사용한다. 요소가 문서 흐름의 "땅"에서 빠져나와 position layer에 올라간다고 이해하면 쉽다.

기본값인 `position: static`은 떠 있지 않은 상태다. `relative`, `absolute`, `fixed`는 모두 요소를 position layer로 올리지만, 기준 사각형이 다르다.

## 8. `relative`, `absolute`, `fixed`는 기준 사각형이 다르다

Position을 이해할 때 핵심은 값의 이름보다 기준 사각형이다.

- `relative`: 자기 원래 자리의 사각형을 기준으로 움직인다.
- `absolute`: 기준 표시가 된 조상 요소의 사각형을 기준으로 움직인다.
- `fixed`: viewport의 사각형을 기준으로 움직인다.

`relative`는 원래 있던 자리에 "분신"을 남겨두고, 실제 요소가 그 자리 기준으로 이동하는 것처럼 볼 수 있다. 그래서 주변 흐름을 크게 무너뜨리지 않으면서 기준 표시용으로 자주 쓰인다.

`absolute`는 원하는 조상을 기준으로 삼아 그 사각형 안에서 위치를 잡는다. 기준 표시가 없으면 조상들을 따라 올라가다가 결국 최상위 기준까지 올라간다.

`fixed`는 부모나 조상 흐름과 관계없이 viewport를 기준으로 고정된다. 스크롤해도 상단에 남는 메뉴나 화면 중앙의 알림창은 이 성격을 활용한다.

따라서 position을 쓸 때 첫 질문은 "이 요소가 누구의 사각형을 기준으로 떠야 하는가"이다.

## 9. `absolute`의 기준 표시는 보통 `relative`로 한다

`absolute` 요소는 기준이 되는 조상이 필요하다. 이 기준은 `position: relative`, `position: absolute`, `position: fixed` 등 static이 아닌 position 값으로 표시할 수 있다.

하지만 실무에서는 기준 표시용으로 `relative`가 자주 쓰인다. 이유는 `relative`가 원래 자리의 공간을 유지하면서 position 기준만 만들어주기 때문이다.

```css
.card {
  position: relative;
}

.badge {
  position: absolute;
  top: -10px;
  right: -10px;
}
```

이 패턴은 카드 모서리의 배지, 댓글 아이콘, 알림 점처럼 부모 박스 기준으로 살짝 걸쳐야 하는 요소에 자주 쓰인다.

중요한 것은 "`absolute`를 쓰면 부모에 무조건 `relative`를 준다"가 아니다. 기준을 어떤 조상으로 삼을지 결정하고, 그 기준 표시가 주변 레이아웃을 망치지 않도록 `relative`를 선택하는 경우가 많다는 뜻이다.

## 10. `top`, `right`, `bottom`, `left`는 사면의 관계를 정한다

Position에서 `top`, `right`, `bottom`, `left`는 단순한 X/Y 좌표가 아니다. 기준 사각형의 네 면과 자신의 네 면 사이의 거리를 정하는 값이다.

예를 들어 `top: 10px`은 기준 사각형의 위쪽 면에서 자신의 위쪽 면을 10px 떨어뜨린다는 뜻이고, `left: 10px`은 기준 사각형의 왼쪽 면에서 자신의 왼쪽 면을 10px 떨어뜨린다는 뜻이다.

이 관점이 없으면 `left: 0`과 `right: 0`을 같이 줬을 때 왜 늘어나거나 한쪽 기준이 우선되는지 이해하기 어렵다. `width`가 없으면 양쪽 면에 붙으면서 늘어날 수 있고, `width`가 있으면 충돌하는 조건 사이에서 브라우저의 우선 규칙이 적용된다.

Position 상태에서도 margin은 offset 계산에 개입할 수 있다. 다만 `left: 0`에 `margin-left: 10px`을 섞기보다, 의도가 위치 조정이라면 `left: 10px`처럼 offset 값으로 표현하는 편이 더 명확하다.

`margin: auto`도 position layer에서는 normal flow와 다르게 활용될 수 있다. 사방 offset을 잡고 width/height가 정해져 있으면 남는 공간을 margin auto가 나눠 가져서 가운데 정렬처럼 동작할 수 있다.

```css
.modal {
  position: fixed;
  inset: 0;
  width: 320px;
  height: 180px;
  margin: auto;
}
```

이 방식은 viewport 중앙의 알림창이나 고정 모달을 만들 때 쓸 수 있다. 대신 width와 height가 필요하다는 제약이 있다. 크기가 콘텐츠에 따라 유동적이면 다른 정렬 방식이 더 적합할 수 있다.

## 11. Fixed와 `z-index`는 떠 있는 UI의 우선순위를 만든다

`position: fixed`는 viewport를 기준으로 요소를 띄운다. 그래서 스크롤해도 같은 위치에 남아야 하는 상단 메뉴, 고정 버튼, 알림창에 적합하다.

```css
.menu {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  z-index: 100;
}
```

`left: 0`과 `right: 0`을 함께 주면 viewport의 좌우 면에 붙으면서 가로로 늘어난다. 여기에 높이, 배경색, `z-index`를 조합해 상단 고정 메뉴를 만든다.

Position layer에 여러 요소가 올라오면 어떤 요소가 더 위에 보일지 결정해야 한다. 기본적으로는 마크업에서 뒤에 나온 요소가 더 위에 보일 수 있지만, 고정 메뉴나 모달처럼 우선순위를 명시해야 하는 요소가 있으면 `z-index`를 사용한다.

`z-index`는 큰 숫자를 아무렇게나 주는 속성이 아니라, 떠 있는 요소끼리의 우선순위를 관리하는 규칙이다. 겹침 문제가 생겼을 때 무작정 숫자만 키우기보다, 어떤 요소가 position layer에 올라와 있고 어떤 순서로 겹치는지 먼저 확인해야 한다.

## 12. Flex와 Position은 컴포넌트 실습에서 함께 쓰인다

실습에서는 카드 안의 사진이나 아이콘이 분명 `width`와 `height`를 가지고 있는데도 flex 공간 안에서 줄어드는 상황이 나왔다.

Flex container는 기본적으로 자식들을 한 줄 안에 넣으려고 한다. 이때 flex item은 `flex-shrink: 1`이 기본값이므로, 공간이 부족하면 자신의 명시적 크기를 가지고 있어도 줄어들 수 있다.

```css
.photo {
  flex-shrink: 0;
}
```

사진 썸네일, 고정 크기 아이콘, 아바타처럼 크기를 지켜야 하는 요소에는 `flex-shrink: 0`을 줄 수 있다. 다만 무조건 끄면 다른 텍스트나 컨테이너가 넘칠 수 있으므로, 줄어들면 안 되는 요소와 줄어들어도 되는 요소를 나눠야 한다.

또 댓글 아이콘이나 알림 표시처럼 카드 안에서 살짝 빠져나와야 하는 요소는 flex 흐름에 그대로 두기보다 `absolute`로 빼내고, 부모에 기준 표시를 주는 방식이 적합하다.

Position은 시각적 장식과 상태 표시에는 유용하지만, 실제 콘텐츠 흐름에 참여해야 하는 정보를 함부로 빼내면 접근성과 반응형에서 문제가 생길 수 있다. 배치를 해결하는 대신 문서 흐름에서 빠져나오는 비용을 만든다.

## 13. 반응형 웹은 CSS 픽셀과 viewport 이해에서 시작한다

반응형 웹을 다룰 때 먼저 이해해야 하는 것은 CSS 픽셀과 기기의 물리 픽셀이 다르다는 점이다.

스마트폰의 물리 픽셀은 매우 조밀하다. CSS의 `100px`을 물리 픽셀 100개로 그대로 그리면 화면에서 너무 작게 보일 수 있다. 그래서 브라우저는 CSS 픽셀을 기준으로 화면을 해석하고, viewport 설정을 통해 기기 화면에 맞춰 확대/축소한다.

실무에서 기본적으로 쓰는 viewport meta는 이 문제와 연결된다.

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0" />
```

이 코드는 CSS에서 말하는 viewport width를 실제 기기 폭에 맞추고, 초기 확대 비율을 1로 설정한다. 이 설정이 없으면 모바일에서 전체 페이지가 축소되어 글씨가 지나치게 작게 보일 수 있다.

반응형 웹은 단순히 작은 화면을 위한 CSS 몇 줄이 아니라, CSS 픽셀, viewport, 기기 해상도 차이를 전제로 한 레이아웃 설계다.

## 14. Breakpoint와 Media Query는 구조 변화에 쓴다

화면 폭이 조금씩 커질 때 요소의 폭만 자연스럽게 늘어나는 정도라면 flex나 grid의 유연한 계산만으로도 충분할 수 있다.

하지만 어느 순간 1열이 2열이 되고, 2열이 3열이 되거나, 메뉴 구조 자체가 바뀌어야 한다면 별도의 규칙이 필요하다. 이 지점을 breakpoint라고 한다.

Media query는 특정 조건에서만 적용되는 CSS 블록이다.

```css
body {
  background-color: yellow;
}

@media (min-width: 720px) {
  body {
    background-color: red;
  }
}
```

기본 상태에서는 노란색이고, viewport 폭이 720px 이상이 되면 빨간색으로 바뀐다. 여러 구간을 나누고 싶다면 `min-width`와 `max-width`를 함께 사용할 수 있다.

```css
@media (min-width: 721px) and (max-width: 989px) {
  body {
    background-color: green;
  }
}
```

Breakpoint가 많아질수록 디자인도 코드도 관리가 어려워진다. 화면이 깨질 때마다 새 구간을 추가하면 나중에는 어떤 조건에서 어떤 스타일이 적용되는지 추적하기 힘들어진다.

실무에서는 주요 breakpoint를 3~4개 정도로 잡고, 그 사이의 유연한 변화는 flex, grid, 상대 단위, `max-width` 같은 기본 레이아웃 규칙으로 흡수하는 편이 좋다.

## 15. Media Query 안에서도 Cascade는 그대로 작동한다

Media query는 별도의 마법 공간이 아니다. 조건이 맞으면 그 안의 CSS가 일반 CSS처럼 cascade에 참여한다.

따라서 선택자 우선순위가 다르면 media query 안의 스타일이 예상대로 덮어쓰지 못할 수 있다.

예를 들어 기본 CSS에서 `html body`처럼 더 구체적으로 선택하고, media query 안에서는 `body`만 선택하면 기본 CSS가 계속 이길 수 있다. 조건은 맞았지만 specificity에서 밀리는 것이다.

Media query를 사용할 때는 다음을 확인해야 한다.

- 조건이 실제 현재 viewport 폭과 맞는가
- 기본 CSS보다 뒤에 작성되어 cascade상 나중에 오는가
- 선택자 구체성이 기본 CSS와 충돌하지 않는가
- 여러 media query 범위가 서로 겹치거나 비어 있지 않은가

반응형 CSS가 헷갈리는 이유는 media query 자체보다 cascade, specificity, 작성 순서가 함께 얽히기 때문이다.

## 16. 6일차 실무 감각

핵심은 레이아웃 도구를 상황별로 구분하는 것이다.

- normal flow는 문서의 기본 흐름이다.
- flex는 한 방향 정렬과 작은 컴포넌트 내부 배치에 강하다.
- grid는 행과 열이 있는 반복 구조에 강하다.
- position은 흐름에서 빠져 떠야 하는 요소에 쓴다.
- media query는 특정 화면 폭에서 레이아웃 규칙을 바꿀 때 쓴다.

배치 문제를 만났을 때 중요한 것은 "이 속성을 쓰면 된다"가 아니라, 현재 요소가 어떤 흐름에 있고, 그 흐름에서 벗어나야 하는지, 부모가 어떤 배치 책임을 가져야 하는지 판단하는 것이다.

Grid와 position, media query는 각각 강력하지만 부작용도 있다. Grid는 격자 구조에 맞을 때 좋고, position은 흐름에서 빠지는 비용을 만들며, media query는 조건이 늘어날수록 관리 비용이 커진다.

실무 감각은 배치 도구를 많이 아는 것이 아니라, normal flow에서 시작해 필요한 순간에만 flex, grid, position, media query로 이동하는 것이다.
