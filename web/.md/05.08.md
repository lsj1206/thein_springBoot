# [05.08] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

2일차의 중심 주제는 CSS가 HTML 구조를 상자와 상태로 계산하는 방식을 이해하는 것이다.

브라우저는 DOM과 CSSOM을 결합해 각 요소의 박스, 텍스트, 여백, 상태를 계산한다. 그래서 화면을 그대로 따라 치기보다, 디자인을 보고 "어떤 상자들이 어떤 구조로 들어 있는가"를 머릿속에 그릴 수 있어야 한다.

강의 후반부는 CSS 선택자, 클래스 네이밍, Figma의 프레임과 컴포넌트 개념이 실제 코드 구조와 어떻게 연결되는지로 이어진다.

핵심 관점은 다음 문장으로 정리할 수 있다.

> 디자인을 볼 때 보이는 선과 글자를 그대로 보지 말고, 그 뒤에 있는 상자 구조를 머릿속으로 그려야 한다.

## 2. CSS는 CSSOM을 수정하는 언어다

브라우저는 HTML로 DOM을 만들고, CSS로 CSSOM을 만든다. DOM은 문서의 구조 정보이고, CSSOM은 각 요소를 어떻게 그릴지에 대한 스타일 정보다.

CSS는 CSSOM의 값을 바꾸는 언어라고 볼 수 있다.

```css
p {
  color: red;
  border: 1px solid black;
}
```

위 코드는 `p` 요소의 CSSOM 중 `color`, `border-*` 관련 값을 바꾼다. 개발자 도구의 `Computed` 탭을 보면 브라우저가 실제로 계산한 스타일 값을 확인할 수 있다.

기초 문법보다 중요한 것은 CSS가 "화면에 직접 그리는 명령"이 아니라, 브라우저가 계산할 스타일 정보를 바꾸는 선언이라는 점이다. 이 관점을 가지면 왜 기본값, 상속, 우선순위, shorthand가 모두 계산 결과로 합쳐지는지 이해하기 쉽다.

## 3. 태그는 상자를 만들고 CSS는 참여 방식을 정한다

강의에서 반복해서 강조된 핵심은 "태그 하나는 상자 하나를 만든다"는 관점이다.

상자는 화면에 보일 수도 있고, `head`, `style`, `title`, `meta`처럼 기본 스타일에 의해 보이지 않을 수도 있다. 이 차이는 `display` 속성과 user agent stylesheet를 이해하면 설명할 수 있다.

즉, HTML은 구조를 만들고 CSS는 그 구조가 화면에서 어떤 상자로 참여할지 결정한다.

실무에서는 태그를 추가하기 전에 다음을 생각해야 한다.

- 이 태그가 정보 구조에 필요한가
- 레이아웃을 위한 상자인가
- CSS만으로 해결할 수 있는 표현인가
- 불필요한 DOM을 늘리고 있지는 않은가

## 4. 타이포그래피와 `line-height`

1일차에서 다룬 inline과 line box 개념을 바탕으로, 2일차에서는 `line-height`가 실제 CSS 계산에서 어떤 의미를 가지는지로 이어졌다.

`font-size`는 실제 글자의 시각적 높이와 정확히 같지 않을 수 있다. 폰트마다 글자를 배치하는 기준이 다르기 때문이다.

웹에서 `line-height`는 전통적인 편집 디자인의 줄간격과 다르다. CSS에서는 한 줄을 감싸는 line box의 높이로 이해하는 것이 중요하다.

```css
.box {
  font-size: 12px;
  line-height: 24px;
}
```

이 경우 12px짜리 inline box가 24px짜리 line box 안에 배치된다. 한 줄이면 block box 높이는 24px, 두 줄이면 48px이 된다.

Figma의 줄간격 값과 브라우저 렌더링이 완전히 같지 않을 수 있으므로, 실무에서는 화면마다 임의로 맞추기보다 일관된 타이포그래피 규칙을 먼저 확인해야 한다.

## 5. User Agent Stylesheet와 Reset

브라우저는 자체 기본 CSS를 가지고 있다. 이를 user agent stylesheet라고 한다. 예를 들어 `body`에는 기본 margin이 있고, 제목 태그에는 기본 글자 크기와 margin이 들어간다.

디자인을 정확히 구현할 때는 이 기본값이 방해가 될 수 있으므로 reset을 한다.

```css
* {
  margin: 0;
}
```

실무에서는 margin뿐 아니라 padding, box-sizing, font 관련 기본값까지 포함한 reset CSS를 쓰는 경우가 많다.

다만 reset은 기본값을 없애는 선택이다. 브라우저가 제공하던 여백, focus 표시, 폼 요소 기본 스타일 등을 지울 수 있으므로, 제거한 기본 스타일을 프로젝트 규칙으로 다시 설계해야 한다.

즉, reset은 "초기화하면 끝"이 아니라 일관된 UI 기준을 직접 책임지겠다는 선택이다.

## 6. Box Model과 `box-sizing`

CSS 박스 모델은 바깥에서 안쪽으로 `margin`, `border`, `padding`, `content` 영역을 가진다.

```text
margin
border
padding
content
```

퍼블리싱이나 UI 개발은 결국 디자인을 보고 이 상자들을 구조화해 배치하는 일이다.

block 요소의 기본 `width`는 부모 공간을 가능한 한 채우는 방향으로 계산된다. 반면 `height`는 내부 콘텐츠를 감싸는 방향으로 계산된다.

기본 `box-sizing`은 `content-box`이다. 이 경우 `width`와 `height`는 content 영역만 기준으로 계산되고, padding과 border는 바깥으로 더해진다.

디자이너가 말한 전체 크기와 CSS 계산값을 맞추기 위해 보통 `border-box`를 많이 쓴다.

```css
* {
  box-sizing: border-box;
}
```

`border-box`를 사용하면 `width`, `height`가 content, padding, border를 포함한 전체 크기 기준으로 계산된다.

중요한 것은 `border-box`를 습관처럼 넣는 것이 아니라, 어떤 영역을 기준으로 크기를 맞추려는지 이해하는 것이다. margin은 여전히 width 바깥에 있으므로, 3일차의 가로 스크롤 문제와도 연결된다.

## 7. Shorthand는 계산 결과를 압축해서 쓰는 문법이다

CSS는 긴 속성을 매번 쓰지 않도록 shorthand를 제공한다.

```css
.box {
  border: 1px solid red;
}
```

이 한 줄은 네 방향의 width, style, color 값을 한꺼번에 지정한다. 브라우저 내부에는 `border-left-width`, `border-top-style` 같은 세부 속성이 각각 존재한다.

shorthand는 코드를 짧게 만들지만, 기존 세부 속성을 한 번에 덮어쓸 수 있다. 그래서 복잡한 상태 스타일을 수정할 때는 어떤 세부 속성이 함께 바뀌는지 확인해야 한다.

## 8. Class와 ID

요소를 선택하기 위해 이름을 붙일 수 있다.

- `id`: 한 페이지 안에서 고유해야 한다.
- `class`: 여러 요소에 반복해서 붙일 수 있다.

UI 구조와 재사용성을 생각하면 class 기반 설계가 더 자주 쓰인다.

`id`는 고유한 대상 식별에는 적합하지만, 재사용 가능한 UI 조각을 만들기에는 불리하다. 반면 class는 같은 역할이나 상태를 여러 요소에 반복 적용할 수 있어 컴포넌트 구조와 잘 맞는다.

## 9. 이름을 의미 있게 짓는 것이 구조화의 시작이다

강의에서 강조한 실무 습관 중 하나는 "이름을 의미 있게 짓는 것"이다. `aa`, `bb`, `box1`처럼 이름을 붙이면 구조가 드러나지 않는다.

```html
<section class="service">
  <div class="service-item">...</div>
</section>
```

이름은 단순 장식이 아니라 화면의 의미와 역할을 설명한다. 좋은 이름은 나중에 코드를 읽을 때 구조를 바로 추측하게 만든다.

독학 단계에서는 화면이 잘 나오면 이름을 대충 붙이기 쉽다. 하지만 실무에서는 이름이 곧 유지보수 비용이다. 이름만 보고도 "이 요소가 어떤 영역의 어떤 역할인지" 추측할 수 있어야 한다.

## 10. 선택자는 충분히 구체적으로, 과하게 길지 않게

모든 `h3`를 선택하면 원하지 않는 영역의 제목까지 영향을 받을 수 있다. 특정 영역 안의 `h3`만 선택하려면 구조를 활용한다.

```css
.service-item > h3 {
  margin-bottom: 10px;
}
```

`>`는 직계 자식을 의미한다. 조상을 함께 언급하면 더 구체적으로 선택할 수 있지만, 너무 길어지면 가독성과 재사용성이 떨어진다.

강의에서는 "선을 넘지 않는 선에서 충분히 구체적으로 부르는 것"을 권했다. 실무적으로는 전역 오염을 막을 만큼 구체적이되, 구조가 조금 바뀌었다고 CSS가 전부 깨질 만큼 강하게 묶지는 않는 균형이 필요하다.

## 11. 가상 선택자는 브라우저와 CSS가 약속한 이름이다

전사에서는 "약속된 이름"이라는 표현으로 설명되었다. 보통 CSS에서는 pseudo-class, 즉 가상 선택자라고 부른다.

```css
.service > .service-item:last-child {
  margin-bottom: 0;
}
```

가상 선택자를 쓰면 데이터 개수와 상관없이 마지막 요소의 margin만 자동으로 제거할 수 있다. React 같은 환경에서도 별도 class를 붙이는 로직을 줄일 수 있다.

대표적으로 `:first-child`, `:last-child`, `:hover`, `:active`, `:focus` 같은 상태나 위치 이름이 있다.

핵심은 개발자가 직접 붙인 이름과 브라우저가 약속해 둔 이름을 구분하는 것이다. 이 구분이 되면 CSS가 직접 class를 붙이지 않아도 특정 상태를 선택할 수 있다는 점을 이해하기 쉽다.

## 12. Modifier Class와 재사용 가능한 CSS

상태마다 완전히 다른 class를 만들고 CSS를 복사하면 공통 부분을 수정할 때 여러 곳을 고쳐야 한다.

더 나은 방식은 기본 class를 유지하고 상태 class를 추가하는 것이다.

```html
<div class="service-item best">...</div>
<div class="service-item hot">...</div>
```

```css
.service-item {
  padding: 10px;
}

.service-item.best {
  background-color: pink;
}
```

이 구조에서는 `service-item`이 공통 뼈대를 담당하고, `best`, `hot`은 특정 상태만 담당한다. 버튼을 만들 때도 `.btn`에 공통 속성을 두고, `.large`, `.small`, `.red` 같은 class로 크기나 색상 변형을 더할 수 있다.

이 방식의 장점은 재사용성이다. 대신 class 조합이 늘어나면 어떤 조합이 가능한지 관리해야 한다. 그래서 규모가 커질수록 네이밍 규칙과 디자인 시스템이 중요해진다.

## 13. Figma와 코드 구조의 연결

강의에서는 Figma를 단순 디자인 도구가 아니라 코드 구조와 연결되는 도구로 설명했다.

주요 개념은 코드와 대응된다.

- Frame: HTML의 상자, 컨테이너
- Layer tree: DOM 트리의 부모-자식 구조
- Properties panel: CSSOM 속성을 시각적으로 조작하는 패널
- Auto layout: normal flow나 flex layout과 유사한 자동 배치 개념
- Component: React component, CSS의 재사용 가능한 조각
- Variant: 버튼의 normal, best, hot, large, small 같은 상태

Figma에서 레이어 이름과 구조를 잘 설계하면 AI나 MCP를 통한 코드 생성 품질도 높아진다. 핵심은 "그림을 코드처럼 그려야 한다"는 것이다.

디자인 파일이 정리되어 있지 않으면 코드도 정리되기 어렵다. Figma의 구조는 단순한 시각 자료가 아니라 구현 구조의 초안이 될 수 있다.

## 14. 디자인 시스템과 컴포넌트

Figma의 컴포넌트는 코드의 컴포넌트와 같은 문제를 해결한다. 버튼 원본 컴포넌트 하나를 여러 화면에서 쓰면, 원본 수정이 모든 사용처에 반영된다.

이것은 CSS에서 공통 class를 만들고 modifier class로 변형을 주는 방식과 유사하다.

```text
공통 컴포넌트
-> 크기 변형
-> 색상 변형
-> 상태 변형
```

실무 디자인 시스템은 이런 식으로 많은 상태와 변형을 가진 부품 체계를 만든다.

트레이드오프도 있다. 컴포넌트화는 일관성과 재사용성을 높이지만, 처음 구조를 설계하는 비용이 있다. 작은 일회성 화면까지 과하게 시스템화하면 오히려 속도가 느려질 수 있다.

## 15. Input과 Focus

`input`은 사용자 입력을 받는 요소이고, 클릭하거나 키보드로 접근하면 focus 상태가 된다.

```html
<input type="text" />
```

focus 상태는 `:focus`로 선택한다. 브라우저는 기본 outline을 보여주는데, 이를 제거할 때는 접근성을 위해 대체 focus 스타일을 제공해야 한다.

```css
input:focus {
  outline: none;
  box-shadow: 0 0 0 3px rgba(0, 128, 255, 0.3);
}
```

outline은 border와 다르게 레이아웃 부피에 포함되지 않고 바깥에 덧그려지는 성격이 있다.

실무에서 `outline: none`만 넣고 대체 표시를 만들지 않으면 키보드 사용자는 현재 위치를 알기 어렵다. 따라서 focus 스타일은 단순 미관 문제가 아니라 접근성 문제다.

## 16. Transform과 Transition

hover 상태에서 요소를 움직이거나 키울 때 `transform`과 `transition`을 함께 쓴다.

```css
li {
  transition: transform 0.5s ease;
}

li:hover {
  transform: translateX(10px);
}
```

`transform`은 요소를 변형하고, `transition`은 첫 상태와 마지막 상태 사이의 중간 프레임을 브라우저가 자동으로 만들어준다.

강의에서는 transition을 hover 상태가 아니라 기본 상태에 두는 것이 디버깅과 되돌아오는 애니메이션에 유리하다고 설명했다.

`transition: all`은 편하지만, 의도하지 않은 속성까지 애니메이션 대상이 될 수 있다. 실무에서는 가능한 한 `transform`, `opacity`처럼 필요한 속성을 명시하는 편이 안전하다.

애니메이션은 눈에 잘 띄고 성능에도 영향을 주기 때문에, CSS로 해결할 수 있는 것과 JavaScript가 필요한 것을 구분해야 한다.

## 17. 작업 순서 관점

디자인을 코드로 옮길 때는 바깥에서 안쪽으로 들어가는 방식이 좋다.

1. 전체 덩어리의 의미와 이름을 정한다.
2. 가장 바깥 상자의 margin, padding, border, width, height를 잡는다.
3. 내부를 큰 영역 단위로 나눈다.
4. 각 영역 안의 자식 상자를 배치한다.
5. 반복되는 구조는 class와 component로 묶는다.
6. 공통 규칙과 변형 규칙을 분리한다.
7. 마지막 요소, hover, focus 등 상태 규칙을 pseudo-class로 처리한다.

이 순서는 단순 작업 절차가 아니라 사고 순서다. 먼저 큰 상자의 의미와 관계를 잡고, 그 다음 세부 스타일을 넣어야 CSS가 덜 얽힌다.

## 18. 2일차 실무 감각

2일차에서 반복된 실무 감각은 다음과 같다.

1. CSS는 DOM을 직접 바꾸는 것이 아니라 CSSOM의 계산 값을 바꾸는 언어다.
2. 디자인을 볼 때는 선과 글자보다 상자 구조를 먼저 본다.
3. class 이름은 화면의 의미와 역할을 드러내야 한다.
4. 선택자는 전역 오염을 막을 만큼 구체적이고, 구조 변화에 버틸 만큼 유연해야 한다.
5. modifier class와 컴포넌트는 재사용성을 높이지만, 조합 관리 비용도 만든다.
6. focus, transition, transform은 미관뿐 아니라 접근성, 디버깅, 성능과 연결된다.

정리하면 2일차의 핵심은 다음 한 문장이다.

> CSS 작업은 속성을 외워 붙이는 일이 아니라, 디자인을 상자 구조와 상태 시스템으로 번역하는 일이다.
