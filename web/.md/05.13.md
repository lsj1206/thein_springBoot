# [05.13] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

이날 강의는 전날 만든 디자인 토큰과 작은 컴포넌트 작성 방식을 실제 카드 UI로 확장하는 흐름이었다.

중심 주제는 단순히 카드 모양을 CSS로 따라 만드는 것이 아니라, 디자인 파일에 있는 부품 체계를 코드의 부품 체계로 옮기는 것이다.

강의에서 반복된 판단 기준은 다음과 같다.

- 바깥 여백은 개별 부품이 아니라 부모 컨테이너의 책임으로 보는 경우가 많다.
- 이미지는 `img`로 넣을지 `background-image`로 넣을지 용도와 데이터 흐름을 기준으로 판단한다.
- 버튼처럼 동작 의미가 있는 요소는 시각적 모양보다 HTML 의미를 먼저 본다.
- reset CSS는 고정된 정답이 아니라 반복되는 초기화 비용을 줄이기 위해 프로젝트에 맞게 확장한다.
- Figma의 수치와 프레임 구조를 그대로 믿지 말고, 실제 HTML 구조에서 어떤 요소 사이의 거리인지 확인한다.
- `overflow: hidden`, `width: 100%`, 변수화 같은 편리한 선택도 얻는 것과 잃는 것을 같이 봐야 한다.

즉, 5일차는 "작은 UI를 만드는 기술"보다 "부품을 어떤 책임 단위로 나누고, 어떤 기준으로 HTML/CSS 결정을 내릴 것인가"에 가까운 수업이다.

## 2. 부품의 바깥 여백은 부모가 책임지는 경우가 많다

리스트 안에 들어가는 `item` 컴포넌트를 만들 때, 각 아이템의 내부 구조와 리스트 전체의 간격을 구분해야 한다.

독학할 때는 보이는 모양을 빨리 맞추기 위해 개별 아이템에 `margin`이나 `padding`을 계속 넣기 쉽다. 하지만 아이템 사이의 간격, 리스트의 좌우 여백, 리스트 전체의 위아래 여백은 대개 `ul`, `ol`, `.list` 같은 부모 컨테이너가 책임지는 것이 자연스럽다.

개별 `item`은 다음에 집중한다.

- 아이콘 영역
- 텍스트 영역
- 날짜나 상태 정보
- 내부 정렬
- 클릭 가능한 영역의 크기

반대로 리스트 부모는 다음에 집중한다.

- 리스트 전체의 좌우 여백
- 아이템 사이의 간격
- 리스트가 놓이는 영역의 폭
- 반복되는 아이템의 배치 규칙

이 책임을 섞으면 나중에 같은 `item`을 다른 리스트에 넣을 때 문제가 생긴다. 어떤 화면에서는 간격이 맞지만, 다른 화면에서는 아이템 자체가 불필요한 바깥 여백을 품고 있어서 재사용이 어려워진다.

실무에서는 "부품 내부의 여백인가, 부품들 사이의 관계인가"를 먼저 나눈다. 내부 여백은 컴포넌트가 갖고, 외부 관계는 부모 레이아웃이 갖는 쪽이 더 안전하다.

## 3. 상태 아이콘은 `img`와 `background-image` 중 맥락으로 고른다

강의에서는 리스트 아이템 안의 상태 아이콘을 `img` 태그로 넣을지, CSS의 `background-image`로 넣을지 고민했다.

둘 다 같은 아이콘을 보여줄 수 있지만, 차이는 상태를 어디에서 바꾸는가에 있다. 아이콘이 장식에 가깝고 상태 종류가 적다면 class에 따라 `background-image`를 바꾸는 방식이 단순하다. 반대로 상태값이나 이미지 경로가 데이터, props, 서버 응답과 강하게 연결된다면 `img`의 `src`를 바꾸는 구조가 더 명확할 수 있다.

중요한 것은 "아이콘이니까 무조건 background" 또는 "이미지니까 무조건 img"로 외우지 않는 것이다. 서비스에서 이 값이 어떻게 바뀌고, 누가 관리하며, 보조기술에 어떤 의미로 전달되어야 하는지를 기준으로 고른다.

## 4. 데이터 이미지와 장식 이미지는 다르게 본다

카드 안의 아바타 이미지는 `background-image`보다 `img` 태그가 더 적합하다고 정리했다.

이유는 아바타가 단순 장식이 아니라 사용자 데이터이기 때문이다. 사용자가 프로필 이미지를 등록하면 데이터베이스에 저장되고, 화면에서는 그 이미지 경로를 받아서 뿌리게 된다.

이런 경우에는 React나 서버 템플릿에서 `src`를 갱신하는 구조가 자연스럽다. 반면 배경 이미지는 CSS에 이미지 경로가 들어가므로 사용자마다 달라지는 데이터를 표현하기에는 어색하다.

실무에서는 이미지를 "보이는 파일"로만 보지 말고, 데이터인지 장식인지부터 구분해야 한다. 사용자, 상품, 게시글처럼 정보의 일부이면 `img`가 우선이고, 없어도 의미 전달에 문제가 없는 장식이면 `background-image`를 검토한다.

## 5. 클릭 동작이 있으면 시각 형태보다 `button` 의미를 먼저 본다

카드 안의 하트, 댓글, 더보기 같은 요소는 작고 아이콘처럼 보이지만, 사용자가 누르는 동작을 수행한다면 `button`으로 보는 것이 자연스럽다.

아이콘처럼 보인다고 `span`이나 `div`에 클릭 이벤트만 붙이면 화면은 만들 수 있다. 하지만 HTML 의미, 키보드 접근성, 보조기술 전달, 기본 focus 동작은 약해진다.

다만 모든 버튼이 같은 버튼 컴포넌트 class를 가져야 하는 것은 아니다. 강의에서는 기존 버튼 시리즈가 필요하지 않은 아이콘형 버튼이라면, 그 목적에 맞는 별도 class를 줄 수 있다고 보았다.

중요한 판단은 두 단계다.

1. 이 요소가 사용자의 동작을 받는가.
2. 기존 버튼 컴포넌트 계열에 포함할 만큼 같은 규칙을 공유하는가.

첫 번째는 HTML 의미의 문제이고, 두 번째는 디자인 시스템의 문제다. `button` 태그를 쓰되, 기존 버튼 스타일 세트와 무리하게 합치지 않는 선택도 충분히 가능하다.

## 6. Reset CSS는 반복되는 초기화 비용을 줄이는 장치다

버튼을 만들 때마다 `border: 0`, `background: transparent`, `padding: 0` 같은 초기화를 반복한다면 reset CSS에 넣을 수 있다.

이 관점에서 reset CSS는 인터넷에서 가져와 고정해두는 파일이 아니라, 프로젝트를 진행하면서 반복되는 브라우저 기본 스타일과의 마찰을 줄이는 파일이다.

브라우저는 `button`에 기본 border, background, padding 등을 준다. UI를 직접 설계한 디자인으로 맞출 때 이 초기화가 반복된다면 reset에 올리는 것이 합리적이다.

다만 reset은 브라우저가 제공하던 기본 시각 단서도 함께 지울 수 있다. 예를 들어 버튼의 기본 배경과 테두리를 없애면 클릭 가능한 요소라는 시각적 신호가 약해진다. focus 스타일까지 지우면 키보드 사용자는 현재 위치를 잃는다.

따라서 reset에 넣을 때는 반복적으로 제거하는 기본 스타일인지, 제거한 뒤 프로젝트의 버튼 상태 표현을 다시 제공하는지 확인해야 한다. reset은 작업을 편하게 만들지만, 브라우저 기본값의 안전장치까지 함께 제거할 수 있다.

## 7. Flex는 큰 레이아웃뿐 아니라 작은 부품 내부 정렬에도 쓴다

이번 카드 실습에서는 큰 페이지 레이아웃보다 작은 컴포넌트 내부에서 `display: flex`를 반복해서 사용했다.

예를 들어 아바타 이미지와 사용자 이름을 가로로 놓고 세로 가운데를 맞추는 경우, 인라인 요소의 기본 흐름에 기대면 어느 정도 나란히 보일 수 있다. 하지만 정밀하게 제어하기 어렵다.

```css
.index {
  display: flex;
  align-items: center;
  gap: 8px;
}
```

이미지, 텍스트, 숫자, 아이콘이 섞인 작은 UI에서는 "나란히 보인다"와 "정렬 규칙을 갖고 있다"가 다르다.

인라인 흐름은 텍스트 baseline을 기준으로 배치되기 때문에, 이미지와 텍스트의 시각적 중심을 맞추기 어렵다. `flex`를 쓰면 배치 방향과 반대 축 정렬을 명시할 수 있다.

`flex`는 한 번만 쓰는 큰 기술이 아니라, UI의 작은 층마다 필요한 정렬 규칙을 부여하는 도구로 보는 것이 좋다.

## 8. `justify-content`와 `align-items`는 축을 기준으로 판단한다

강의에서는 `justify-content`와 `align-items`를 계속 축 기준으로 설명했다.

`flex-direction: row`라면 주축은 가로이고, 교차축은 세로다. 이때 `justify-content`는 가로 방향의 분배를 다루고, `align-items`는 세로 방향의 정렬을 다룬다.

```css
.index-more {
  display: flex;
  justify-content: space-between;
  align-items: center;
}
```

위 코드에서 왼쪽 사용자 정보와 오른쪽 피드백 영역을 양끝으로 벌리는 것은 주축 분배이므로 `justify-content: space-between`이 맞다. 두 묶음을 세로 가운데 맞추는 것은 교차축 정렬이므로 `align-items: center`가 맞다.

독학 과정에서는 속성을 "가운데 정렬할 때 쓰는 것"처럼 외우기 쉽다. 하지만 실제로는 방향이 바뀌면 같은 속성의 효과도 바뀐다.

예를 들어 `flex-direction: column`이면 주축이 세로가 되므로, 텍스트 묶음을 세로 방향으로 가운데에 놓는 데 `justify-content: center`를 사용할 수 있다.

속성 이름을 외우는 것보다 먼저 확인할 것은 현재 flex container의 주축이 어디인지이다.

## 9. `width: auto`와 padding은 무조건 `box-sizing` 문제로 보지 않는다

강의 후반에는 padding을 줬는데도 `box-sizing: border-box`를 따로 주지 않는 상황이 나왔다.

핵심은 `width`를 명시했는지 여부다.

블록 요소의 `width` 기본값은 `auto`다. 이 상태에서 좌우 padding을 주면 브라우저는 부모의 사용 가능한 공간 안에서 자동으로 content width를 계산한다.

이 경우 `width: 100%`를 굳이 추가하면 오히려 문제가 생길 수 있다. `width: 100%`는 content box의 폭을 부모 폭과 같게 잡고, 여기에 padding이 더해질 수 있기 때문이다. 물론 전역으로 `box-sizing: border-box`를 적용했다면 결과가 달라지지만, 원리를 모르고 습관적으로 `width: 100%`를 붙이는 것은 위험하다.

실무에서는 "꽉 차게 만들고 싶다"는 이유로 `width: 100%`를 남발하지 않는 것이 중요하다. 이미 블록 요소가 `width: auto`로 부모 공간에 맞춰 계산되는 상황이라면 기본값을 활용하는 편이 더 안전하다.

## 10. Figma 수치는 실제 HTML 구조 기준으로 다시 읽어야 한다

Figma에서 간격을 잴 때는 어떤 프레임과 어떤 요소 사이를 재고 있는지 주의해야 한다.

디자인 파일에는 시각적 정리를 위해 여러 프레임이 들어가고, Auto Layout이나 그룹 구조가 코드의 HTML 구조와 1:1로 대응하지 않을 수 있다. 그래서 Figma에서 보이는 `gap`이나 거리값을 그대로 CSS에 옮기면 실제 화면과 어긋날 수 있다.

이번 실습에서도 아이콘과 숫자 사이, 버튼 사이, 아바타와 텍스트 사이의 간격을 하나씩 확인했다. 중요한 것은 "Figma에서 13px이라고 보인다"가 아니라, 실제 코드에서 어떤 요소 사이의 간격이 13px이어야 하는지이다.

특히 디자인 도구의 프레임은 개발자가 만드는 DOM 요소와 목적이 다를 수 있다. Figma 구조를 참고하되, CSS는 실제 HTML 책임 단위에 맞춰 작성해야 한다.

## 11. 카드의 `overflow: hidden`은 반복 class를 줄이지만 잘림을 만든다

카드 안에 사진 컴포넌트가 들어갈 때, 사진이 카드의 위쪽에 있으면 위쪽 모서리만 둥글어야 하고, 아래쪽에 있으면 아래쪽 모서리만 둥글어야 한다. 이를 모두 class로 나누면 `.photo.top`, `.photo.middle`, `.photo.bottom`처럼 경우가 늘어난다.

강의에서는 더 단순한 방법으로 카드 자체에 `border-radius`와 `overflow: hidden`을 주는 방식을 사용했다. 이렇게 하면 카드 밖으로 넘치는 사진 영역이 잘리므로, 사진 위치마다 모서리 class를 따로 만들 필요가 줄어든다.

```css
.card {
  border-radius: var(--round-m);
  overflow: hidden;
}

.photo img {
  width: 100%;
  display: block;
}
```

하지만 `overflow: hidden`은 말 그대로 넘치는 것을 잘라낸다. 내부 요소의 그림자, focus ring, 툴팁, 드롭다운, 뱃지처럼 카드 밖으로 나와야 하는 요소도 함께 잘릴 수 있다.

따라서 이 선택은 모서리 처리 class를 줄이는 대신, 카드 경계 밖으로 보여야 하는 요소까지 자를 수 있다. `overflow: hidden`은 편리한 해결책이지만, 시각적 잘림을 만드는 강한 제약이기도 하다.

## 12. 이미지 아래 여백은 inline baseline 때문에 생길 수 있다

카드 안의 사진 아래에 예상하지 못한 공백이 생겼다. 원인은 이미지가 기본적으로 inline 요소처럼 baseline 위에 놓이기 때문이다.

인라인 formatting context에서는 글자의 descender 영역을 고려하기 때문에, 이미지 아래에도 텍스트 기준선과 관련된 여분의 공간이 남을 수 있다.

이 문제는 이미지 파일이나 카드 높이 문제가 아니라 브라우저의 인라인 배치 원리 때문에 생기는 공백이다. 해결은 이미지를 block으로 바꾸는 것이다.

```css
.photo img {
  display: block;
  width: 100%;
}
```

`display: block`이 되면 이미지는 더 이상 텍스트 baseline 위에 올라가는 인라인 요소처럼 배치되지 않는다. 그래서 아래 공백이 사라진다.

flex container 안에서는 inline 요소가 block처럼 다뤄져 같은 문제가 가려질 수 있다. 하지만 "flex를 주면 해결된다"로 외우면 원인을 놓친다. 이미지 아래 애매한 공백이 보이면 먼저 baseline 문제를 의심해야 한다.

## 13. CSS는 HTML 구조 순서와 책임 단위를 따라 작성한다

카드 실습에서도 CSS는 HTML 구조를 따라가며 작성했다.

먼저 큰 카드 박스를 만들고, 그 안의 `index`, `feedback`, `photo` 같은 하위 부품으로 내려갔다. 이 방식은 단순히 보기 좋게 정리하는 문제가 아니라, CSS를 읽는 사람이 DOM 구조와 스타일 책임을 함께 추적할 수 있게 한다.

예를 들어 `.card` 안에 `.index-more`, `.index`, `.feedback`, `.photo`가 있다면 CSS도 큰 카드에서 하위 부품으로 내려가는 순서가 읽기 쉽다.

반대로 시각적으로 눈에 보이는 순서나 작업한 순서대로 CSS를 흩어놓으면, 나중에 어떤 규칙이 어떤 부품에 속하는지 찾기 어려워진다.

실무에서는 CSS 자체도 문서다. HTML 구조, 컴포넌트 책임, 스타일 규칙의 위치가 어느 정도 같은 방향을 바라보게 정리해야 유지보수 비용이 줄어든다.

## 14. 디자인 컴포넌트 체계를 코드 컴포넌트 체계로 옮긴다

강의 중반에는 디자이너도 Figma 안에서 버튼, 사진, 카드 같은 부품을 만들어 재사용한다는 관점이 나왔다.

과거에는 Photoshop 중심 작업처럼 디자인 규칙이 코드와 잘 맞물리지 않는 경우가 많았다. 그러나 Figma는 컴포넌트와 Auto Layout을 통해 디자인 안에서도 재사용 가능한 체계를 만들 수 있게 했다.

프론트엔드 작업은 이 디자인 체계를 코드 체계로 옮기는 일이다. 버튼은 버튼 컴포넌트로, 색상과 간격은 CSS 변수와 토큰으로, 카드 부품은 HTML 구조와 class 체계로 연결된다.

최근에는 AI나 디자인-to-code 도구가 이 연결을 자동화하려고 한다. 그렇기 때문에 오히려 개발자는 더 명확한 판단 기준이 필요하다. 도구가 생성한 코드가 디자인 부품의 의도, HTML 의미, 접근성, 유지보수 구조를 제대로 반영했는지 검토해야 하기 때문이다.

즉, 목표는 "Figma를 눈대중으로 베끼기"가 아니다. 디자인 시스템의 규칙을 코드에서도 재사용 가능한 규칙으로 바꾸는 것이다.

## 15. 공통 class를 억지로 만들지 않는다

비슷해 보이는 요소가 있다고 해서 반드시 공통 class로 묶어야 하는 것은 아니다.

강의에서는 아바타가 들어가는 타입과 더보기 버튼이 들어가는 타입이 시각적으로 비슷한 위치에 있을 수 있지만, 공통점이 충분하지 않다면 같은 class로 묶지 않는 판단을 했다.

공통 class는 중복을 줄이지만, 잘못 묶으면 서로 다른 목적의 요소가 같은 규칙에 묶인다. 이후 한쪽만 수정해야 할 때 다른 쪽까지 영향을 받는다.

공통화할 때는 같은 구조, 같은 의미, 같은 상태 변화, 같은 변경 방향을 공유하는지 확인해야 한다. 단지 현재 화면에서 위치나 크기가 비슷해 보인다는 이유만으로 묶으면 나중에 분리 비용이 생긴다.

공통화는 "같아 보이기 때문에"가 아니라 "같은 이유로 변하기 때문에" 해야 한다.

## 16. 토큰화할 값과 지역 값을 구분한다

카드 실습에서는 라운드, 그림자, 회색 계열 같은 값은 변수로 가져와 사용했다. 이는 디자인 시스템 안에서 반복될 가능성이 높기 때문이다.

반면 어떤 빨간색 값은 변수로 만들지 않고 직접 쓰는 판단이 나왔다. 이유는 그 색이 디자인 시스템의 의미 있는 색인지, 디자이너가 의도적으로 만든 상태 색인지, 아니면 단발성 수치인지 확실하지 않았기 때문이다.

변수는 많이 쓰면 좋지만, 모든 수치를 무조건 변수로 빼는 것이 정답은 아니다.

반복되고 의미가 명확하며 테마나 유지보수 대상이 될 값은 변수화하기 좋다. 반대로 한 곳에서만 쓰이거나 디자인 의도가 불명확한 값은 지역 값으로 남겨도 된다.

실무에서는 애매한 값을 억지로 변수화해 그럴듯한 이름을 붙이는 것이 오히려 위험하다. 의미가 없는 변수 이름은 나중에 시스템을 흐리게 만든다.

## 17. 측정값은 `gap`, `padding`, `margin` 중 관계에 맞게 넣는다

카드 안에서 10px, 11px, 13px 같은 간격을 맞추는 과정이 있었다. 중요한 것은 숫자 자체가 아니라 그 숫자가 어떤 관계를 표현하는지이다.

부모 박스의 좌우 안쪽 여백이라면 `padding`이 맞고, 아바타와 이름 사이처럼 형제 요소 사이의 거리라면 `gap`이 맞다. 컴포넌트와 다른 컴포넌트 사이의 외부 관계라면 부모 레이아웃의 `gap`이나 필요한 경우 `margin`을 검토한다.

독학할 때는 거리값을 맞추기 위해 아무 요소에나 `margin-left`를 넣는 경우가 많다. 하지만 그런 방식은 코드가 쌓일수록 관계를 읽기 어렵게 만든다.

거리값을 넣기 전에 먼저 질문해야 한다.

- 이 거리는 요소 내부의 여백인가
- 형제 요소 사이의 간격인가
- 부모와 자식 사이의 관계인가
- 컴포넌트 바깥의 배치 관계인가

CSS 속성은 숫자를 넣는 곳이 아니라 관계를 표현하는 도구다.

## 18. 5일차 실무 감각

5일차의 핵심은 작은 UI 하나를 만들면서도 계속 판단을 분리하는 것이다.

- 이 여백은 부품 내부인가, 부모와의 관계인가
- 이 이미지는 데이터인가, 장식인가
- 이 요소는 클릭 동작이 있는가
- 이 값은 디자인 토큰인가, 지역 값인가
- 이 간격은 `gap`, `padding`, `margin` 중 무엇으로 표현해야 하는가
- 이 편한 해결책은 어떤 부작용을 만드는가

코드 자체는 `display: flex`, `gap`, `padding`, `width: 100%`, `overflow: hidden`처럼 익숙한 속성의 반복이다. 하지만 실제 난이도는 속성 암기가 아니라, 어떤 요소에 어떤 책임을 줄지 결정하는 데 있다.

이날 강의는 카드 하나를 만드는 실습처럼 보이지만, 실제로는 디자인 시스템을 코드 시스템으로 옮길 때 필요한 판단 기준을 반복해서 훈련한 수업이다.
