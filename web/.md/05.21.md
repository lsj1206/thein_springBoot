# [05.21] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

JavaScript는 HTML/CSS로 만든 문서와 스타일 위에서 값을 계산하고, 실행 흐름을 만들고, DOM 상태를 바꾼다.

반드시 잡아야 할 기준은 다음과 같다.

- 표현식은 값을 만들고, 문은 실행할 일을 지시한다.
- 변수와 함수는 값이나 코드 조각에 의미를 붙이는 추상화 수단이다.
- 프로그램은 평가 단계에서 실행 환경을 준비하고, 실행 단계에서 실제 값을 대입하거나 함수를 호출한다.
- 함수는 값처럼 전달될 수 있고, 다른 함수의 반환값이 될 수도 있다.
- 이벤트 처리는 함수를 즉시 실행하는 것이 아니라, 브라우저에게 나중에 실행할 일을 등록하는 방식이다.
- `==`의 자동 형 변환과 참조값 비교는 JavaScript에서 버그를 만들기 쉬운 지점이다.

핵심은 "값이 언제 만들어지고, 어디에 저장되며, 어떤 시점에 실행되는가"를 기준으로 코드를 읽는 것이다.

## 2. JavaScript는 값을 만들고 의미를 붙인다

JavaScript 코드는 여러 실행 단위로 나뉜다. 식(expression)은 값을 만들고, 문(statement)은 실행할 일을 지시한다. 블록(block)은 여러 문을 묶고, 함수(function)는 재사용 가능한 실행 단위를 만들며, 프로그램(program)은 전체 실행 흐름을 구성한다.

표현식은 값을 만들지만 그 값이 자동으로 보이는 것은 아니다. `3 * 5 + 10 / 2`는 계산 결과를 만들 수 있지만, `console.log`, `alert`, DOM의 `textContent`처럼 결과를 전달받는 대상이 없으면 화면이나 콘솔에 드러나지 않는다.

프로그래밍은 작은 값에 의미를 붙이고 다시 조합하는 과정이다. 같은 숫자를 반복하는 것보다 `pi`, `radius`, `area`처럼 역할을 가진 이름을 두면 값의 의도와 변경 지점이 드러난다. 변수는 가장 기본적인 추상화 수단이고, 함수는 여러 문장을 묶어 이름을 붙이는 더 큰 추상화 수단이다.

좋은 추상화는 코드를 짧게 만드는 것보다 의미를 안정적으로 만든다. 이름만 보고 의도를 추측할 수 있어야 수정 범위와 버그 위치를 좁히기 쉽다.

## 3. 평가 단계는 실행 환경을 준비한다

JavaScript 엔진은 코드를 단순히 위에서 아래로 한 줄씩만 처리하지 않는다. 먼저 평가 단계에서 실행 컨텍스트와 렉시컬 환경을 준비하고, 이후 실행 단계에서 실제 대입과 호출을 진행한다.

브라우저 전역 코드가 실행되면 Global 실행 컨텍스트가 콜스택에 올라가고, Global 렉시컬 환경이 만들어진다. 이 안에는 브라우저가 제공하는 전역 객체와 연결되는 환경, 코드에서 선언한 식별자를 관리하는 환경이 함께 잡힌다. `window`, `alert`, `console`, `log` 같은 이름은 전자에 가깝고, 코드에서 새로 선언한 이름은 후자에서 관리된다.

`var b = { 나이: 20 }`은 처음부터 `b`가 객체를 가진 상태로 시작한다고 보기보다, 평가 단계에서 `b`라는 이름을 준비하고 실행 단계에서 객체를 만들어 연결하는 흐름으로 보는 편이 정확하다. 객체 자체는 별도 메모리 영역에 만들어지고, 변수에는 그 객체를 가리키는 참조가 들어간다.

`this` 바인딩과 Outer 환경 참조도 실행 환경을 준비하는 과정에 포함된다. 그래서 전역의 `this`가 브라우저에서 `window`를 가리킨다는 사실만 외우기보다, 실행 컨텍스트가 어떤 `this`와 외부 환경 참조를 들고 있는지로 읽는 편이 안전하다.

`undefined`, 호이스팅, TDZ 같은 현상은 문법 예외가 아니라 평가 단계와 실행 단계에서 무엇이 먼저 준비되는지의 차이로 설명된다.

## 4. 스코프는 렉시컬 환경 체인으로 값을 찾는다

렉시컬 환경은 변수 이름을 실제 값으로 연결하기 위한 구조다. 코드가 식별자를 만나면 현재 실행 컨텍스트의 환경 레코드에서 먼저 찾고, 없으면 Outer 환경 참조를 따라 바깥 렉시컬 환경으로 이동한다.

```js
var b = 30;
let a = 100;

if (a > 20) {
  let a = 10;
  console.log(a); // 10
  console.log(b); // 30
}
```

`if` 블록에 들어가면 Block 렉시컬 환경이 만들어지고, 그 안에 새로운 `a`가 준비된다. 그래서 `console.log(a)`는 Global의 `a = 100`이 아니라 가장 가까운 Block 환경의 `a = 10`을 찾는다.

반면 `console.log(b)`를 실행할 때는 Block 환경 안에 `b`가 없다. 이때 엔진은 Outer 환경 참조를 따라 Global 렉시컬 환경으로 이동하고, 그곳에서 `b = 30`을 찾는다.

함수 호출도 이 구조 안에서 일어난다. `console.log(a)`가 실행되면 `log` 함수의 실행 컨텍스트가 콜스택에 올라가고, 실행이 끝나면 해당 실행 컨텍스트는 제거된다. 더 이상 참조되지 않는 환경은 정리 대상이 된다.

블록 스코프의 핵심은 "밖에서 접근할 수 없다"가 아니라, 실행 중에 어떤 환경부터 이름을 찾고 언제 그 환경이 사라지는지까지 포함한다. 실무에서는 이 원리를 이름 충돌을 줄이고 영향 범위를 좁히는 기준으로 쓴다.

## 5. 함수 호출은 반환값과 부수효과를 구분해서 읽는다

함수는 매개변수로 값을 받고, 내부 문장을 실행하고, 필요하면 결과를 반환한다. 어떤 함수 `sum`이 있다면 이름만 쓴 `sum`은 함수 값 자체를 가리키고, `sum(1)`은 함수를 실행한 결과를 뜻한다. 호출 위치에는 반환값이 남고, 그 값을 변수에 저장하거나 다른 표현식의 일부로 사용할 수 있다.

반환값이 있는 함수를 호출만 하고 사용하지 않으면 계산 결과는 다음 흐름에서 활용할 수 없다. 반대로 화면 출력, DOM 수정, 콘솔 출력처럼 부수효과가 목적이면 반환값을 저장하지 않아도 된다.

함수의 목적이 "값을 계산해 돌려주는 것"인지, "실행 과정에서 외부 상태를 바꾸는 것"인지 구분해야 한다. 이 구분이 흐려지면 함수 이름만 보고도 사용법을 예측하기 어렵다.

## 6. 함수는 값처럼 전달되고 반환될 수 있다

JavaScript에서 함수는 값처럼 변수에 담기고, 다른 함수의 인자로 전달될 수 있다.

`b`가 매개변수 `fn`으로 받은 함수를 내부에서 호출하는 구조라면, `b(a)`는 `a` 함수를 즉시 실행해서 넘기는 것이 아니라 함수 값 자체를 전달한다. 반대로 `b(a())`는 `a`를 먼저 실행한 결과를 넘긴다. `b` 내부에서 `fn()`을 호출하는 순간에야 전달받은 함수의 코드가 실행되며, 이 구조가 callback의 기본 감각이다.

함수의 반환값도 함수일 수 있다.

```js
function c() {
  return function () {
    console.log("반갑습니다.");
  };
}

c()();
```

`c()`를 실행하면 내부 함수가 반환되고, 뒤의 `()`는 반환된 그 함수를 다시 실행한다. `c()();`는 "함수를 실행해서 함수를 얻고, 얻은 함수를 다시 실행한다"는 뜻이다.

고차 함수, 커링, 이벤트 핸들러 생성, 라이브러리 API는 `()`가 어느 함수에 붙어 있고 각 호출의 반환값이 무엇인지 끊어서 읽어야 한다.

## 7. IIFE와 FD/FE는 평가와 실행 관점에서 읽는다

IIFE는 Immediately Invoked Function Expression의 약자로, 함수 표현식을 만든 즉시 실행하는 패턴이다.

```js
const result = (function (value) {
  return value + 10 + 10;
})(1);
```

앞의 괄호는 `function` 선언을 표현식으로 다루게 만들고, 뒤의 `(1)`은 그 함수 표현식을 즉시 호출한다. IIFE는 특이한 암기 문법이 아니라, 함수가 값이고 표현식이 될 수 있다는 사실에서 나온 패턴이다.

Function Declaration과 Function Expression도 평가 단계에서 다르게 보인다.

```js
const result = sum(1, 2);

function sum(num1, num2) {
  return num1 + num2;
}

const result2 = sum2(1, 2);
const sum2 = function (num1, num2) {
  return num1 + num2;
};
```

함수 선언문은 평가 단계에서 함수 객체까지 준비되므로 선언문보다 앞에서 호출해도 동작할 수 있다. 반면 함수 표현식은 변수에 함수 값을 대입하는 방식이므로, 대입 전에 호출하면 안전하지 않다.

실무에서는 선언 방식의 취향보다 호출 순서와 가독성이 중요하다. 함수가 파일 전체에서 공통 동작으로 쓰이면 선언문이 읽기 쉬울 수 있고, 특정 값처럼 전달하거나 조건부로 만들 함수라면 표현식이 자연스러울 수 있다.

## 8. DOM API는 객체를 찾아 반환한다

DOM 조작 코드는 `document.querySelector(...).addEventListener(...)` 형태만 암기하면 흐름을 놓치기 쉽다.

`querySelector`는 브라우저가 제공하는 함수이고, 선택자에 맞는 DOM 객체를 찾아 반환한다. 반환된 객체를 `buttonEl` 같은 변수에 저장하면 이후 그 객체의 속성이나 메서드를 사용할 수 있다.

`addEventListener`도 버튼 객체가 가진 메서드를 호출하는 것이다. DOM API도 결국 함수 호출, 반환값, 객체 속성 접근이라는 JavaScript 기본 원리 위에서 읽어야 한다.

실무에서는 `querySelector`가 항상 요소를 찾는다고 가정하면 위험하다. 선택자가 틀리거나 DOM이 아직 만들어지지 않은 시점이면 `null`이 될 수 있으므로, 필요한 경우 존재 여부를 확인해야 한다.

## 9. 이벤트 처리는 나중에 실행할 함수를 등록하는 일이다

이벤트 처리는 "버튼을 누르면 함수가 실행된다"보다 한 단계 더 정확하게 이해해야 한다. 브라우저에게 특정 사건이 발생했을 때 실행할 함수를 등록하는 일이다.

```js
const buttonEl = document.querySelector("#btn");

buttonEl.addEventListener("click", clickWork);

function clickWork() {
  const inputEl = document.querySelector("#name");
  const resultEl = document.querySelector("#result");

  resultEl.textContent = `${inputEl.value}님 반갑습니다.`;
  resultEl.classList.add("red");
}
```

`clickWork`는 등록 시점에 실행되지 않는다. 클릭 이벤트가 발생하면 브라우저가 그 함수를 호출한다. `addEventListener('click', clickWork())`처럼 괄호를 붙이면 함수가 즉시 실행되고, 이벤트에는 실행 결과가 전달될 수 있다.

사용자가 입력한 값을 읽고 화면에 반영하는 과정은 DOM 객체의 속성을 읽고 쓰는 과정이다. `inputEl.value`는 입력값을 읽고, `resultEl.textContent`는 요소 안의 텍스트를 바꾸며, `classList.add('red')`는 기존 class를 유지한 채 필요한 class를 추가한다.

핵심은 JavaScript가 HTML/CSS와 분리된 별도 세계가 아니라는 점이다. JS는 DOM 객체의 상태를 바꾸고, 브라우저는 바뀐 DOM/CSSOM 기준으로 다시 화면을 계산한다.

## 10. 자료형은 값의 사용 규칙을 정한다

JavaScript 값은 종류마다 사용할 수 있는 연산과 비교 규칙이 다르다. 변수 선언 시점에 타입을 강하게 고정하지 않고, 실행 중 들어 있는 값에 따라 동작이 결정된다. TypeScript는 이 느슨함을 보완하기 위해 타입을 더 엄격하게 적게 하는 도구로 볼 수 있다.

불리언 값인 `true`, `false`는 조건 판단의 결과값으로 자주 쓰인다. 객체는 이름이 붙은 값의 묶음이고, 배열은 순서가 있는 값의 묶음이다. JavaScript 배열은 사용 관점에서는 배열처럼 다루지만, 연속 메모리 배열과 같은 성능/메모리 모델을 그대로 보장한다고 보면 안 된다.

`undefined`와 `null`은 모두 "값이 없음"을 표현하지만 뉘앙스가 다르다. `undefined`는 시스템이 아직 값이 없다고 만든 상태에 가깝고, `null`은 개발자가 의도적으로 비어 있음을 넣는 값으로 쓰는 경우가 많다.

## 11. 비교는 자동 형 변환과 참조값 때문에 조심한다

JavaScript에는 느슨한 비교인 `==`와 엄격한 비교인 `===`가 있다. `==`는 비교 전에 타입을 맞추려고 자동 형 변환을 시도한다.

```js
"82" == 82; // true
"82" === 82; // false
```

이런 친절함은 간단한 예제에서는 편해 보일 수 있지만, 실무에서는 버그의 원인이 되기 쉽다. 일반적인 업무 코드에서는 `===`를 기본 선택으로 두고, 타입 변환이 필요하면 비교 전에 명시적으로 변환하는 편이 의도를 드러낸다.

객체, 배열, 함수는 값의 내용이 아니라 참조를 비교한다.

```js
{} === {}; // false
[] === []; // false

const a = {};
const b = a;

a === b; // true
```

두 객체의 내용이 비어 있어도 각각 새로 만들어졌다면 서로 다른 참조를 가진다. 반대로 같은 객체의 참조를 다른 변수에 복사하면 두 변수는 같은 대상을 가리킨다.

이 차이는 상태 관리, 배열 복사, 객체 비교, React 같은 UI 라이브러리의 렌더링 판단에서도 중요하다. "내용이 같아 보인다"와 "같은 참조를 가리킨다"는 완전히 다른 판단이다.

## 12. 7일차 실무 감각

JavaScript를 DOM 조작 예제의 나열로 보지 않는다.

먼저 표현식이 값을 만들고, 문이 실행 흐름을 만들며, 함수가 그 흐름을 재사용 가능한 단위로 묶는다는 점을 봐야 한다. 그 다음 평가/실행 구분으로 변수, 함수, 호이스팅, 스코프를 읽어야 한다.

DOM 이벤트 코드는 이 기본 원리 위에 올라간다. `querySelector`는 객체를 반환하고, `addEventListener`는 함수를 나중에 실행하도록 등록하며, `textContent`와 `classList`는 DOM 객체의 상태를 바꾼다.

마지막으로 JavaScript의 자료형과 비교 규칙은 실무 버그를 만들기 쉽다. `===`를 기본으로 쓰고, 객체/배열/함수는 내용이 아니라 참조를 비교한다는 점을 기준으로 코드를 읽어야 한다.

핵심은 "코드가 어떤 값을 만들었는가"와 "그 값이 언제 실행되거나 사용되는가"를 끝까지 추적하는 것이다.
