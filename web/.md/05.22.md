# [05.22] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

JavaScript는 조건과 반복을 통해 실행 흐름을 만들고, 객체와 배열을 통해 화면과 데이터의 상태를 유연하게 다룬다.

반드시 잡아야 할 기준은 다음과 같다.

- 표현식은 값을 만들고, 문은 실행 흐름을 만든다.
- `const`는 변수 binding을 고정하지만 객체 내부 property 변경까지 막지는 않는다.
- truthy/falsy는 조건 판단의 기준이 되며, 값의 비어 보이는 정도와 항상 일치하지 않는다.
- `&&`, `||`는 Boolean만 반환하지 않고 실제 operand 값을 반환할 수 있다.
- short-circuit은 최종값뿐 아니라 오른쪽 표현식의 실행 여부까지 바꾼다.
- 반복문은 종료 조건과 갱신을 함께 설계해야 하며, 브라우저 main thread를 오래 점유할 수 있다.
- 객체 property와 배열은 실제 자료 구조의 성격에 맞는 접근 도구를 선택해야 한다.

핵심은 "어떤 값이 만들어졌는가", "그 값이 어떤 조건에서 선택되는가", "어떤 객체나 배열 도구를 사용할 수 있는가"를 기준으로 코드를 읽는 것이다.

## 2. 표현식은 값을 남기고 문은 흐름을 만든다

표현식(expression)은 실행된 뒤 값이 남는다. 그래서 함수 인자, 변수 할당, property 할당, `return` 뒤처럼 값이 필요한 자리에 들어갈 수 있다.

```js
resultEl.textContent = count > 0 ? `${count}개` : "없음";
```

삼항 연산자는 조건에 따라 둘 중 하나의 값을 남기므로 할당식 안에서 자연스럽게 쓰인다. 위 코드는 `count > 0`의 결과에 따라 DOM 객체의 `textContent`에 들어갈 문자열을 선택한다.

반대로 `if`는 값을 남기는 도구라기보다 실행 흐름을 나누는 문(statement)이다. 한 줄의 값 선택이면 삼항 연산자가 간결하고, 여러 작업이 분기마다 달라지면 `if / else if / else`가 흐름을 더 잘 드러낸다.

기준은 짧은 코드가 아니라 역할이다. 값 선택이면 표현식으로 압축하고, 실행 경로 자체가 중요하면 문으로 남긴다.

## 3. `const`는 binding을 고정한다

`const`는 변수 이름이 다른 값으로 재할당되는 것을 막는다. 객체를 담은 `const`도 마찬가지로, 변수에 연결된 참조 자체를 바꾸는 것은 막지만 객체 내부 property 변경까지 막지는 않는다.

```js
const user = { name: "홍길동", age: 30 };

user = {}; // 불가능
user.name = "김철수"; // 가능
```

`user`라는 이름을 다른 객체로 다시 연결하는 것은 막힌다. 하지만 `user.name`은 같은 객체 안의 property를 바꾸는 일이므로 허용된다.

객체 내부까지 변경되지 않아야 한다면 `Object.freeze()` 같은 별도 도구를 검토해야 한다. 다만 객체를 완전히 얼리면 수정 가능성이 줄어드는 만큼 사용성도 떨어진다.

실무에서는 기본적으로 `const`를 쓰고, 값이 실제로 다시 연결되거나 카운터처럼 계속 바뀌어야 할 때만 `let`을 쓴다. 객체 내부 변경까지 금지할지는 데이터의 중요도와 유지보수 비용을 보고 별도로 판단한다.

## 4. truthy/falsy는 조건 판단의 기준이다

JavaScript 조건식은 값 자체를 Boolean처럼 평가한다. 이때 참처럼 평가되는 값을 truthy, 거짓처럼 평가되는 값을 falsy라고 볼 수 있다.

주의해야 할 falsy 값은 다음과 같다.

- `false`
- `0`
- `""`
- `null`
- `undefined`
- `NaN`

빈 객체 `{}`와 빈 배열 `[]`은 비어 보여도 truthy다. 따라서 "내용이 비어 보이는가"와 "조건에서 참으로 평가되는가"는 분리해서 봐야 한다.

```js
Boolean({}); // true
Boolean([]); // true
Boolean(""); // false
Boolean(null); // false
```

조건 판단 결과가 반드시 Boolean이어야 한다면 `Boolean(value)`나 `!!value`처럼 의도를 드러내는 편이 안전하다.

## 5. 논리 연산자는 값을 선택한다

JavaScript의 `&&`, `||`는 조건 판단 뒤 Boolean만 남기지 않는다. truthy/falsy 판단을 하면서 실제 operand 값을 반환할 수 있다.

```js
const displayName = userName || "익명";
const adminLabel = isAdmin && "관리자";
```

`||`는 왼쪽이 truthy면 왼쪽 값을 그대로 반환하고, 왼쪽이 falsy면 오른쪽 값을 반환한다. 그래서 기본값 처리에 자주 쓰인다.

`&&`는 왼쪽이 falsy면 왼쪽 값을 그대로 반환하고, 왼쪽이 truthy면 오른쪽 값을 반환한다. 그래서 조건부 실행이나 조건부 렌더링 패턴에서 자주 보인다.

이 패턴은 짧고 강력하지만 반환값이 Boolean인지 실제 데이터인지 헷갈리기 쉽다. 조건 결과만 필요한 코드와 값 선택이 필요한 코드를 구분해서 써야 한다.

## 6. short-circuit은 실행 여부까지 바꾼다

논리 연산자는 왼쪽 값만으로 결과를 결정할 수 있으면 오른쪽 표현식을 실행하지 않는다. 이를 short-circuit으로 볼 수 있다.

```js
const value = cachedValue || createDefaultValue();
```

`cachedValue`가 truthy이면 `createDefaultValue()`는 호출되지 않는다. 이 패턴은 기본값 생성 비용을 줄일 수 있지만, 오른쪽 함수에 부수효과가 있다면 실행 여부가 조건에 묶인다.

우선순위도 값 흐름에 영향을 준다.

```js
const result1 = a || (b && c);
const result2 = (a || b) && c;
```

두 코드는 괄호 위치에 따라 먼저 평가되는 묶음이 달라진다. `a`, `b`, `c`가 Boolean이 아닌 값이면 최종적으로 남는 값도 달라질 수 있다.

실무에서는 우선순위를 암기해 압축하기보다 괄호와 변수 이름으로 의도를 드러내는 편이 안전하다.

## 7. 객체는 동적 property 저장소에 가깝다

JavaScript 객체는 property 이름을 문자열처럼 다루고, 실행 중에 어떤 key로 접근할지 결정할 수 있다.

```js
const user = {
  name: "홍길동",
  age: 30,
  job: "도적",
};

user.name = "김철수";
user["age"] = 25;

function resetProperty(prop) {
  user[prop] = undefined;
}

resetProperty("job");
```

점 표기법은 property 이름이 코드 작성 시점에 고정되어 있을 때 읽기 좋다. 대괄호 표기법은 property 이름을 변수나 표현식으로 계산해야 할 때 필요하다.

`resetProperty(prop)`에서 `prop`에 `"name"`, `"age"`, `"job"` 중 무엇이 들어오든 같은 코드 한 줄로 접근할 수 있다. 문자열 key와 동적 property 접근을 이용하면 긴 분기문을 줄이고, 데이터가 어떤 property를 가리킬지 runtime에 결정하게 만들 수 있다.

단, 동적 key는 오타와 예상하지 못한 key에도 열려 있다. 그래서 입력값 검증, 허용 key 목록, 기본값 처리가 함께 필요하다.

## 8. 배열은 method와 prototype을 가진 객체 계열 값이다

JavaScript 배열은 순서 있는 값의 묶음이지만, 단순한 순서 목록으로만 끝나지 않는다.
배열은 method와 prototype을 가진 객체 계열 값이고, `length`와 index 접근 규칙을 함께 가진다.

```js
const fruits = ["사과", "바나나", "딸기", "수박", "참외"];

fruits.indexOf("참외"); // 4
typeof fruits; // "object"
Array.isArray(fruits); // true
```

`indexOf`는 배열이 가진 method다. 이 method를 쓰려면 대상이 실제 배열이어야 한다.

`typeof []`는 `"array"`가 아니라 `"object"`를 반환한다. JavaScript의 역사적 설계 때문에 배열도 `typeof`로는 객체처럼 보인다. 배열 여부가 중요하면 `Array.isArray()`를 사용해야 한다.

## 9. 배열처럼 보이는 객체와 실제 배열은 다르다

숫자 key와 `length`를 가진 객체는 배열처럼 보일 수 있다.

```js
const fakeArray = {
  0: "사과",
  1: "바나나",
  2: "딸기",
  length: 3,
};

fakeArray[2]; // "딸기"
Array.isArray(fakeArray); // false
```

`fakeArray[2]`처럼 index 접근이 되므로 배열처럼 느껴질 수 있다. 하지만 실제 배열이 아니므로 배열 method를 가진다고 보장할 수 없다. `fakeArray.indexOf("딸기")` 같은 코드는 실패한다.

실무에서는 API 응답, DOM API 반환값, 다른 라이브러리 결과처럼 외부에서 온 데이터를 자주 다룬다.
이때 "배열처럼 접근된다"와 "배열 method를 사용할 수 있다"는 다른 판단이다.

객체와 배열은 서로 중첩될 수 있다.

```js
const student = {
  name: "홍길동",
  hobbies: ["독서", "운동", "게임"],
  address: {
    city: "서울",
    district: "강남구",
  },
};

student.hobbies[2]; // "게임"
student.address.district; // "강남구"
```

복합 데이터는 한 번에 읽으려고 하면 복잡하다. 객체 property로 들어가고, 배열 index로 들어가고, 다시 객체 property로 들어가는 식으로 접근 경로를 끊어서 읽어야 한다.

## 10. 8일차 실무 감각

JavaScript 코드는 값 선택, 실행 경로, 반복 흐름, property 접근, 배열 판별이 함께 얽힌다.

`const`는 binding을 고정하고, 객체 내부 property 변경은 별도의 문제로 남는다. truthy/falsy는 조건 판단의 기준이 되고, `&&`, `||`는 실제 값을 선택하는 도구가 될 수 있다. short-circuit은 오른쪽 표현식의 실행 여부까지 바꾼다.

객체는 runtime key로 접근할 수 있고, 배열은 method와 prototype을 가진 객체 계열 값이다. 배열처럼 보이는 객체와 실제 배열을 구분해야 배열 method를 안전하게 사용할 수 있다.

실무에서 중요한 것은 문법을 많이 아는 것이 아니라, 값이 언제 선택되고, 어떤 참조가 유지되며, 어떤 객체/배열 도구를 사용할 수 있는지 판단하는 것이다.
