# [05.26] 웹퍼블리싱 및 UI개발 기초

## 1. 강의의 큰 관점

JavaScript는 값을 직접 계산하는 것에서 끝나지 않고, 값이 가진 내장 method를 꺼내 쓰고, DOM 객체를 찾아 상태를 바꾸고, 이벤트 흐름에 맞춰 함수를 실행한다.

반드시 잡아야 할 기준은 다음과 같다.

- 배열 method는 원본을 바꾸는지, 새 값을 만들어 반환하는지부터 구분한다.
- DOM 선택 결과가 단일 객체인지 여러 객체 묶음인지에 따라 접근 방식이 달라진다.
- CSS 표현은 JavaScript로 직접 쓰기보다 class 상태를 바꾸고 CSS가 표현을 맡게 하는 쪽이 유지보수에 유리하다.
- DOM 생성은 노드 객체를 직접 조립하는 방식과 HTML 문자열을 삽입하는 방식을 구분해서 선택한다.
- 이벤트는 등록된 함수만 실행하는 것이 아니라, 이벤트 객체와 전파 흐름까지 함께 남긴다.
- HTML/Figma 구조는 CSS 선택자뿐 아니라 JavaScript listener 개수와 이벤트 위임 가능성까지 바꾼다.

핵심은 "어떤 객체가 어떤 도구를 제공하고, 그 도구가 원본과 화면 상태를 어떻게 바꾸는가"를 기준으로 읽는 것이다.

## 2. 값마다 내장 도구가 붙어 있다

JavaScript의 객체, 배열, 함수, 문자열 같은 값은 각자 다루기 위한 method를 가진다. 직접 만든 함수가 아니어도 `arr.concat(...)`, `text.split(...)`, `element.addEventListener(...)`처럼 점을 찍어 도구를 꺼내 쓸 수 있다.

이 도구들은 값의 종류에 따라 다르다. 배열에는 배열을 다루는 method가 있고, 문자열에는 문자열을 다루는 method가 있으며, DOM 객체에는 화면 구조와 속성을 다루는 method가 있다.

문자열이나 숫자처럼 단순해 보이는 값도 method를 호출하는 순간 JavaScript가 필요한 도구를 사용할 수 있게 처리한다. 그래서 `text.length`, `text.substring(...)`, `text.indexOf(...)` 같은 접근이 가능하다.

실무에서는 method 이름만 외우기보다 "이 값이 어떤 종류의 값이고, 지금 꺼낸 method가 어떤 책임을 갖는가"를 먼저 봐야 한다. 같은 점 접근이라도 배열 method인지, DOM method인지, CSSOM을 건드리는 속성인지에 따라 부작용이 달라진다.

## 3. 배열 method는 원본을 바꾸는지부터 구분한다

배열 method를 읽을 때 가장 먼저 볼 것은 반환값보다 원본 변경 여부다. `concat`은 두 배열의 값을 복사해 새 배열을 만들고, 기존 배열은 그대로 둔다. 원본 데이터가 이후에도 필요하다면 이런 비파괴 method가 안전하다.

```js
const merged = first.concat(second);
const copied = list.slice(1, 4);
const removed = list.splice(1, 2);
```

`slice`는 지정한 범위를 복사해 새 배열을 반환하고 원본을 바꾸지 않는다. 반면 `splice`는 지정한 위치에서 요소를 제거하거나 끼워 넣으며 원본을 바꾼다. 전사에서는 두 표현이 섞여 들릴 수 있지만, 실무 기준은 "원본이 남는가"로 구분하면 된다.

`indexOf`는 값을 찾으면 index를 반환하고, 없으면 `-1`을 반환한다. 이 `-1`은 "검색 실패"라는 의미 있는 값이므로 조건문에서 명확히 다뤄야 한다.

`push`, `pop`, `unshift`, `shift`도 원본 배열을 바꾼다. `push`는 뒤에 추가하고 새 길이를 반환하며, `pop`은 뒤에서 하나를 꺼내 반환한다. `unshift`는 앞에 추가하고, `shift`는 앞에서 하나를 꺼낸다.

## 4. 앞쪽 배열 조작은 비용과 의도를 함께 본다

배열의 뒤쪽에 값을 추가하거나 제거하는 `push`, `pop`은 비교적 단순하다. 이미 있는 요소들의 index를 다시 밀어야 하는 일이 적기 때문이다.

반대로 `unshift`, `shift`, 중간 삽입/삭제는 기존 요소들의 index를 다시 맞춰야 한다. 데이터가 몇 개 없을 때는 체감되지 않지만, 수만 개 단위의 데이터를 반복해서 앞쪽에서 넣고 빼면 비용이 커질 수 있다.

JavaScript 배열이 저수준의 연속 메모리 배열과 완전히 같지는 않더라도, 앞쪽 요소를 바꾸면 뒤의 index 관계를 다시 정리해야 한다는 점은 여전히 중요하다.

실무에서는 배열 method를 "동작한다"로 끝내지 않고, 데이터 양과 사용 패턴까지 함께 본다. 로그처럼 뒤에 계속 쌓는 데이터는 `push`가 자연스럽고, 큐처럼 앞에서 계속 꺼내야 하는 구조는 반복적인 `shift` 대신 별도 index를 두거나 자료구조를 분리하는 판단이 필요하다.

## 5. 여러 DOM 객체는 querySelectorAll과 반복으로 다룬다

`querySelector`는 조건에 맞는 첫 번째 DOM 객체 하나를 반환한다. 같은 선택자에 맞는 요소가 여러 개 있어도 첫 번째만 다룬다.

여러 요소를 한 번에 다뤄야 하면 `querySelectorAll`을 사용한다. 반환값은 여러 DOM 객체를 담은 `NodeList`이므로, 하나의 element처럼 바로 `style`이나 `classList`를 붙일 수 없다.

```js
const items = document.querySelectorAll("#service li");

for (let i = 0; i < items.length; i += 1) {
  items[i].classList.add("active");
}
```

처음에는 `items[0]`, `items[1]`처럼 직접 접근할 수 있지만, 요소 개수가 바뀔 때마다 코드를 고치는 구조가 된다. 반복문과 `length`를 사용하면 DOM 개수가 늘거나 줄어도 같은 흐름으로 처리할 수 있다.

여기서 중요한 것은 반복문 자체가 아니라 "선택 결과가 여러 개라면 각 DOM 객체를 하나씩 꺼내서 다뤄야 한다"는 감각이다.

## 6. JavaScript로 CSS를 직접 쓰기보다 class 상태를 바꾼다

JavaScript에서 `element.style.backgroundColor = "lightblue"`처럼 직접 스타일을 넣을 수 있다. 이 접근은 DOM 객체의 `style` 속성을 통해 CSSOM에 영향을 주는 방식이다.

하지만 표현 규칙이 JavaScript 안에 많이 들어오면 CSS가 맡아야 할 책임과 JS가 맡아야 할 책임이 섞인다. 색, 테두리, radius 같은 표현은 CSS에 두고, JavaScript는 상태 class만 붙이는 쪽이 유지보수에 유리하다.

```css
.service-item.active {
  border: 1px solid red;
  background-color: lightblue;
}
```

```js
items[i].classList.add("active");
items[i].textContent = `${items[i].textContent} 서비스`;
```

이 구조에서는 JavaScript가 "활성 상태가 되었다"는 사실만 DOM에 표시하고, 실제 표현은 CSS가 결정한다. 나중에 디자인을 바꿔도 JavaScript 로직을 건드릴 이유가 줄어든다.

실무 기준은 간단하다. 화면 상태를 바꾸는 조건 판단과 데이터 반영은 JavaScript가 담당하고, 그 상태가 어떤 색과 모양으로 보일지는 CSS가 담당하게 나누는 편이 안전하다.

## 7. HTML 속성은 DOM 객체에서 필요한 데이터로 읽힌다

HTML에 적힌 속성은 DOM 객체를 통해 다시 읽을 수 있다. 예를 들어 `title`, `href` 같은 값은 단순한 표시 정보가 아니라 JavaScript가 사용할 수 있는 데이터가 되기도 한다.

```js
const listEl = document.querySelector("#service ul");
const titleValue = listEl.getAttribute("title");
```

`getAttribute`는 HTML에 적힌 속성 값을 문자열로 가져온다. 여러 `a` 태그의 `href`를 읽어서 배열에 모으거나, DOM에 심어 둔 설명 값을 로직에서 사용할 때 이 흐름을 쓴다.

다만 모든 데이터를 HTML 속성에 넣어두는 것이 좋은 설계라는 뜻은 아니다. 속성은 마크업과 연결된 가벼운 데이터에 적합하고, 복잡한 상태나 업무 데이터는 별도 데이터 구조로 관리하는 편이 낫다.

반복문 안에서 할 일도 구분해야 한다. `li`마다 해야 할 작업은 반복문 안에 두고, `ul`에서 한 번만 읽으면 되는 속성은 반복문 밖에서 처리한다.

## 8. DOM 생성은 노드 생성 방식과 문자열 삽입 방식을 구분한다

DOM을 새로 만들 때는 `createElement`, `setAttribute`, `appendChild`로 노드 객체를 하나씩 조립할 수 있다.

```js
const liEl = document.createElement("li");
const linkEl = document.createElement("a");

linkEl.setAttribute("href", href);
linkEl.textContent = label;
liEl.appendChild(linkEl);
listEl.appendChild(liEl);
```

이 방식은 각 노드와 속성을 명시적으로 다루기 때문에 구조가 분명하다. 대신 단순한 HTML 조각을 만들 때도 코드가 길어지고, 부모/자식 관계를 직접 이어 붙여야 한다.

`insertAdjacentHTML`은 부모나 기준 요소 주변에 HTML 문자열을 삽입한다. `beforebegin`, `afterbegin`, `beforeend`, `afterend`처럼 어느 위치에 넣을지 먼저 정하고, 두 번째 인자로 넣을 HTML 문자열을 전달한다.

```js
listEl.insertAdjacentHTML(
  "beforeend",
  `<li><a href="${href}">${label}</a></li>`
);
```

이 방식은 짧고 읽기 쉽지만, 문자열로 HTML을 만들기 때문에 신뢰할 수 없는 입력을 그대로 넣으면 보안 문제가 생길 수 있다. 사용자가 입력한 값이나 외부 API 응답을 넣어야 한다면 escaping, 검증, 텍스트 삽입 방식까지 함께 고려해야 한다.

## 9. className보다 classList가 class 단위 조작에 맞다

`className`은 요소의 class 문자열 전체를 다룬다. `element.className = "active"`처럼 대입하면 기존 class가 사라질 수 있고, 기존 class를 보존하려면 문자열 조합을 직접 신경 써야 한다.

class는 공백으로 나뉜 token들의 집합에 가깝다. 그래서 class 하나를 추가하거나 제거할 때는 `classList`가 더 자연스럽다.

```js
titleEl.classList.add("active");
titleEl.classList.remove("active");
titleEl.classList.toggle("active");
```

`classList.contains("active")`는 특정 class가 있는지 확인하고, `add`, `remove`, `toggle`은 class 하나를 단위로 다룬다. 문자열 전체를 덮어쓰지 않기 때문에 기존 class와 충돌할 가능성이 줄어든다.

실무에서는 단순히 class 이름을 바꾸는 문제가 아니라 상태를 안정적으로 표현하는 문제로 봐야 한다. 활성/비활성, 열림/닫힘, 선택됨/선택 안 됨처럼 UI 상태는 class token으로 표현하고, CSS는 그 token을 기준으로 화면을 그리는 구조가 좋다.

## 10. 이벤트 객체는 사건의 정황을 전달한다

`addEventListener`에 등록한 함수는 이벤트가 발생했을 때 브라우저가 호출한다. 이때 브라우저는 사건의 정황을 담은 이벤트 객체를 첫 번째 인자로 전달한다.

```js
linkEl.addEventListener("click", function (event) {
  event.preventDefault();
  event.target.classList.add("active");
});
```

이벤트 객체에는 클릭 위치, 눌린 대상, 기본 동작 제어 method 같은 정보가 들어 있다. `event.target`은 실제 이벤트가 발생한 요소를 가리키고, `preventDefault()`는 브라우저가 원래 하려던 기본 동작을 막는다.

예를 들어 `a` 태그는 클릭하면 `href`로 이동하려는 기본 동작을 가진다. JavaScript로 화면 일부만 바꾸려는 상황에서 이 기본 동작을 그대로 두면 페이지 이동이나 새로고침 때문에 방금 바꾼 화면이 사라질 수 있다.

이벤트 handler를 읽을 때는 "어떤 사건에 반응하는가", "브라우저 기본 동작을 막아야 하는가", "실제로 눌린 대상이 무엇인가"를 함께 확인해야 한다.

## 11. 이벤트 전파는 listener 위치와 개수를 결정한다

DOM 이벤트는 실제 대상에서만 끝나지 않는다. 브라우저는 바깥쪽에서 안쪽으로 내려오는 capturing 단계와, 대상에서 다시 바깥쪽으로 올라가는 bubbling 단계를 가진다.

이 흐름 덕분에 여러 자식 요소마다 listener를 모두 붙이지 않고, 공통 부모에 listener 하나를 붙여 처리할 수 있다. 이를 이벤트 위임(event delegation) 관점으로 볼 수 있다.

```js
listEl.addEventListener("click", function (event) {
  const linkEl = event.target.closest("a");

  if (!linkEl) {
    return;
  }

  event.preventDefault();
  linkEl.classList.toggle("active");
});
```

`event.target`은 실제 클릭된 요소이고, `event.currentTarget`은 listener가 붙어 있는 요소다. 이벤트 위임에서는 이 차이를 알아야 부모에서 받은 이벤트를 실제 자식 동작으로 연결할 수 있다.

포트폴리오나 실무 코드에서 카드 100개, 썸네일 100개에 각각 listener를 붙이는 방식은 불필요한 비용이 될 수 있다. 꼭 개별 listener가 필요한 경우가 아니라면, 공통 부모와 이벤트 전파를 이용해 listener 개수를 줄이는 구조를 먼저 검토한다.

## 12. HTML/Figma 구조는 JavaScript 구현 비용을 바꾼다

JavaScript 코드를 깔끔하게 만들려면 HTML 구조가 먼저 맞아야 한다. 공통 부모가 없으면 이벤트 위임을 걸 위치가 애매해지고, 의미 없이 흩어진 구조는 CSS 선택자와 DOM 탐색을 복잡하게 만든다.

마크업은 모양만 맞추는 일이 아니다. 어떤 요소들을 하나의 그룹으로 묶을지, 어느 요소가 상태 class를 가져야 하는지, listener를 어디에 붙일 수 있는지까지 결정한다.

Figma 구조도 같은 영향을 준다. AI가 Figma를 보고 React나 HTML 구조를 만들 때, 디자인 레이어가 의미 있는 그룹으로 잡혀 있지 않으면 코드도 불필요하게 복잡해지기 쉽다.

실무에서는 "보이는 화면"과 "코드에서 다룰 구조"를 동시에 본다. CSS가 선택하기 쉬운 구조, JavaScript가 적은 listener로 제어할 수 있는 구조, 상태 class가 자연스럽게 붙을 수 있는 구조가 좋은 HTML 구조다.

## 13. 9일차 실무 감각

배열 method는 이름보다 원본 변경 여부를 먼저 본다. `concat`, `slice`처럼 새 값을 만드는 method와 `splice`, `push`, `pop`, `shift`, `unshift`처럼 원본을 바꾸는 method는 코드의 안정성과 데이터 재사용 가능성을 다르게 만든다.

DOM 조작은 선택 결과의 형태를 기준으로 읽는다. `querySelector`는 하나의 DOM 객체를 반환하고, `querySelectorAll`은 여러 객체 묶음을 반환한다. 여러 요소를 다룰 때는 hard coding된 index보다 반복문과 `length`를 기준으로 처리한다.

표현 변경은 JavaScript가 직접 CSS 값을 쓰는 방식보다 class 상태를 바꾸는 방식이 낫다. JavaScript는 상태와 데이터 흐름을 담당하고, CSS는 그 상태가 어떻게 보일지를 담당하게 나누면 수정 범위가 작아진다.

이벤트 코드는 handler 등록만 보는 것이 아니라 이벤트 객체, 기본 동작, target, 전파 흐름까지 함께 봐야 한다. 특히 listener 개수는 성능과 구조에 영향을 주므로, 공통 부모와 이벤트 위임을 사용할 수 있는 HTML 구조를 먼저 생각해야 한다.

핵심은 JavaScript method를 외우는 것이 아니라, 값과 DOM과 이벤트가 어떤 책임을 나눠 갖는지 보고 구조를 선택하는 것이다.
