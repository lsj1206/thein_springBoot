# 강의 압축 기준

## 강의 주제

웹퍼블리싱 및 UI개발 기초

## 압축 목표

녹화 전사본을 단순히 짧게 줄이는 것이 아니라, 강의에서 반복된 핵심 관점과 실무 판단 기준을 읽기 쉬운 요약본으로 재구성한다.

최종 문서는 나중에 다시 읽었을 때 다음을 바로 떠올릴 수 있어야 한다.

- 강의의 큰 주제가 무엇이었는지
- 각 개념이 왜 필요한지
- 브라우저, CSS, JavaScript 실행 환경이 실제로 어떻게 계산하고 실행하는지
- 실무에서 어떤 판단을 해야 하는지
- 어떤 선택이 무엇을 얻고 무엇을 희생하는지

## 대상 독자 기준

요약본은 HTML/CSS를 처음 배우는 사람을 위한 완전 기초 설명서가 아니다.

대상 독자는 HTML/CSS를 독학했고, 개인 프로젝트를 진행해본 경험이 있다. 따라서 태그, class, CSS 속성 작성법 같은 기본 문법 설명은 길게 다루지 않는다.

JavaScript 내용은 새로 다루더라도 초보자용 문법 암기표처럼 정리하지 않는다. 식과 문, 스코프, 평가와 실행, 함수 값, 반환값, DOM API, 이벤트 등록 시점처럼 독학자가 예제 코드만 따라 하면서 놓치기 쉬운 실행 원리를 우선한다.

대신 독학 과정에서 놓치기 쉬운 부분을 우선해서 남긴다.

- 브라우저 내부 동작: DOM, CSSOM, render tree, layout, paint
- JavaScript 실행 모델: expression, statement, scope, evaluation/execution, function value, event timing
- 예상과 다르게 동작하는 CSS의 원인
- 실무에서 더 안전한 구조와 선택 기준
- 성능, 유지보수, 접근성, 의미 구조 관점의 주의점
- 표면적인 해결책이 만드는 부작용
- AI나 검색으로 가져온 코드를 검토할 기준

즉, "무엇을 쓰는가"보다 "왜 그렇게 동작하는가", "언제 조심해야 하는가", "무엇을 기준으로 선택해야 하는가"를 더 중요하게 다룬다.

## 기본 문서 구조

문서 제목은 날짜와 강의명을 함께 쓴다.

```md
# [05.11] 웹퍼블리싱 및 UI개발 기초
```

첫 번째 섹션은 항상 강의 전체의 관점을 정리한다.

```md
## 1. 강의의 큰 관점
```

이 섹션에서는 그날 강의가 이전 강의와 어떻게 연결되는지, 중심 주제가 무엇인지, 반복해서 강조된 태도가 무엇인지 정리한다.

이후 섹션은 전사 순서를 그대로 따르기보다 개념 흐름을 기준으로 재배치한다.

마지막 섹션은 `N일차 실무 감각` 형식으로 통일하고, 그날의 판단 기준을 짧게 회수한다.

## 자료 반영 우선순위

전사본은 기본 자료로 사용하되, 그대로 옮기지 않는다. 말버릇, 반복, 실습 대기 시간, 전사 오류를 제거하고 개념 흐름으로 재구성한다.

강의에 사용된 HTML, PDF, Figma export, 별도 메모 같은 보조 자료는 본문에 그대로 붙여 넣지 않는다. 대신 다음 목적에만 사용한다.

- 전사본에서 흐릿한 개념의 실제 예시를 확인할 때
- 다이어그램이 강조한 실행 순서나 구조를 보강할 때
- 누락된 전사 구간의 핵심 내용을 복원할 때
- 코드 예시의 정확한 형태를 확인할 때

전사 누락 구간이 있고 사용자가 핵심 메모를 제공한 경우, 그 메모는 반드시 반영한다. 단, 별도 섹션으로 억지로 끼우기보다 해당 개념 흐름 안에 자연스럽게 흡수한다.

보조 자료가 있다고 해서 문서가 자료 해설서가 되어서는 안 된다. PDF나 Figma 다이어그램은 개념의 강조점을 확인하는 근거로 쓰고, 요약본에는 나중에 다시 읽을 판단 기준만 남긴다.

## 섹션 작성 방식

각 섹션은 다음 흐름을 기본으로 한다.

1. 개념을 필요한 만큼만 짧게 정의한다.
2. 독학자가 오해하기 쉬운 지점을 짚는다.
3. 브라우저의 계산 원리나 실무 맥락을 설명한다.
4. 필요한 경우 짧은 코드나 예시를 넣는다.
5. 마지막에 실무 판단, 주의점, 트레이드오프를 덧붙인다.

개념만 나열하지 말고, 원리와 판단 기준까지 남긴다.

이미 알고 있을 가능성이 높은 기초 문법은 줄인다. 예를 들어 `class`가 무엇인지보다, class를 어떤 기준으로 이름 붙이고 재사용 가능한 구조로 나누는지가 더 중요하다.

## 독자 수준과 서술 범위

사용자가 특정 선행 지식을 제시하면, 그 지식을 바탕으로 새로 판단해야 할 원리와 주의점을 중심으로 정리한다.

본문에는 압축 판단 자체를 드러내지 않고, 남은 개념 자체를 서술한다.

JavaScript 요약에서는 표면 문법보다 다음처럼 실행 의미가 뚜렷한 지점을 우선한다.

- 동적 타입과 자동 형 변환
- truthy/falsy와 `&&`, `||`의 operand 반환
- `const` binding과 객체 내부 mutation
- 객체의 동적 property 접근
- 배열의 method, prototype, `Array.isArray()`
- 브라우저 main thread를 막는 긴 동기 실행 비용

날짜별 요약본은 독립적으로 읽혀야 한다. 선행 지식 용어는 압축 기준을 잡기 위한 보조 장치로만 사용하고, 본문에서는 현재 학습 중인 기술의 특징으로 서술한다.

## 구조 압축과 섹션 밀도 기준

섹션은 개념을 잘게 나누는 것보다, 함께 판단해야 하는 흐름끼리 묶는다.

특히 전사본이 길거나 JavaScript처럼 새 개념이 많이 등장하는 날에는 세부 개념별로 섹션을 나누면 튜토리얼처럼 보이기 쉽다. 이때는 다음처럼 넓은 판단 단위로 묶는다.

- 기초 실행 단위: 식, 문, 블록, 함수, 프로그램, 추상화
- 실행 모델: 평가, 실행, 실행 컨텍스트, 렉시컬 환경, 스코프
- 함수 사용 방식: 호출, 반환값, callback, 함수 반환, IIFE, FD/FE
- 브라우저 연결: DOM 선택, 이벤트 등록, DOM 속성 변경
- 위험 지점: 자동 형 변환, 참조값 비교, 부작용, 유지보수 비용

목표는 모든 키워드를 빠짐없이 제목으로 세우는 것이 아니라, 나중에 읽을 때 "어떤 문제를 어떤 기준으로 판단해야 하는가"가 보이게 만드는 것이다.

기존 날짜 파일과 밀도를 맞춘다. 섹션 수는 보통 10~16개를 기준으로 하되, 단일 주제가 선명한 날은 8~12개, 실습이나 여러 축이 섞인 날은 17~19개 안팎까지 허용한다.

대규모 재구성이 필요한 경우 먼저 묶음 기준을 정한다. 예를 들어 `05.21`처럼 JavaScript 첫 수업은 `값과 실행 단위`, `평가와 스코프`, `함수 값`, `DOM 이벤트`, `타입과 비교`처럼 넓은 흐름으로 잡고, 그 안에서 필요한 예시만 남긴다.

## 반복 개념 처리 기준

같은 개념이 여러 날짜에 반복해서 나오면 중요도가 높은 주제일 수 있으므로, 해당 날짜에서 필요한 판단만 남긴다.

다만 파일끼리 강하게 결합되지 않도록 본문에서 "앞에서 설명했듯이", "05.11 참고"처럼 다른 날짜 파일을 직접 참조하지 않는다. 각 날짜 요약본은 혼자 읽어도 핵심 판단이 이해되어야 한다.

반복 개념은 다음 방식으로 줄인다.

1. 해당 파일 안에서 필요한 원리를 1~3문장으로만 다시 요약한다.
2. 긴 배경 설명, 계산 과정, 이미 나온 코드 예시는 반복하지 않는다.
3. 그날 강의에서 새로 등장한 적용 상황, 판단 기준, 트레이드오프만 남긴다.
4. 새 관점이 거의 없으면 별도 섹션으로 두지 않고 실무 감각이나 인접 섹션에 흡수한다.

예를 들어 `img`와 `background-image`의 차이가 다시 나오면 두 방식의 전체 원리를 길게 반복하지 않는다. 대신 그날 실습에서 사용자 아바타, 상태 아이콘, 장식 이미지 중 무엇을 어떤 기준으로 골랐는지만 남긴다.

목표는 각 파일의 독립성을 유지하면서도 반복 설명의 길이를 줄이는 것이다.

## 압축 시 남길 내용

다음 내용은 우선적으로 남긴다.

- 강사가 반복해서 강조한 관점
- HTML, CSS, JavaScript 실행 모델, 브라우저 렌더링과 직접 연결되는 핵심 개념
- 실무에서 버그나 오해를 만들기 쉬운 지점
- 독학으로는 지나치기 쉬운 브라우저 내부 동작
- 성능, 유지보수, 접근성, 의미 구조와 관련된 판단 기준
- 어떤 방식을 선택했을 때 얻는 점과 잃는 점
- "이렇게 하면 된다"보다 "왜 이렇게 해야 하는지"에 해당하는 설명
- 실무에서 자주 쓰지만 부작용이 있는 해결책의 주의점

## 압축 시 줄일 내용

다음 내용은 요약본에서 길게 다루지 않는다.

- 쉬는 시간, 잡담, 전사 오류
- 같은 설명의 반복
- 강의 흐름상 의미가 약한 말버릇
- 핵심 개념 없이 길게 이어지는 예시
- 코드 전체를 이해하는 데 필요하지 않은 장황한 코드
- 나중에 읽어도 판단 기준을 주지 않는 세부 상황 설명
- HTML/CSS 또는 기본 JavaScript 예제를 접해본 사람에게 너무 당연한 기초 문법 설명
- 단순 속성 암기 수준에서 끝나는 설명

단, 반복 설명 중에서도 강사가 강조하려는 핵심 태도가 드러나는 부분은 압축본의 중심 문장으로 남긴다.

## 코드 예시 기준

코드는 개념 이해에 꼭 필요한 만큼만 남긴다.

코드 예시는 다음 목적 중 하나가 있을 때 사용한다.

- 문법 형태를 보여줄 때
- 계산 기준을 설명할 때
- 좋은 예와 나쁜 예를 비교할 때
- 실무에서 자주 쓰는 패턴을 정리할 때
- 독학자가 흔히 잘못 이해하는 지점을 바로잡을 때

코드가 없어도 문장으로 충분히 설명되는 내용은 코드로 늘리지 않는다.

단순 사용법을 보여주는 코드보다, 왜 문제가 생기는지 드러나는 코드가 더 가치 있다.

한 섹션 안에서 비슷한 형태의 코드가 반복되면 대표 예시 하나로 합친다. 특히 기본 문법 확인용 코드, 정상 동작 예시, 같은 결론을 반복하는 출력 예시는 줄인다.

좋은 코드 예시는 다음 중 하나를 드러내야 한다.

- 값이 만들어지는 시점
- 실행 순서
- 기준이 되는 박스나 렉시컬 환경
- 잘못 쓰면 생기는 버그
- 실무에서 선택을 갈라야 하는 지점

코드펜스 수가 많아져 문서가 실습 자료처럼 보이면, 문장 설명으로 충분한 예시는 인라인 코드나 일반 문장으로 바꾼다.

## 문체와 용어 기준

문체는 강의 요약본처럼 쓴다.

- 평어체는 `이다`, `한다`를 기본으로 한다.
- 너무 구어적인 표현은 정리한다.
- 단정적으로 외우게 만들기보다 이유를 설명한다.
- "무조건", "항상" 같은 표현은 실제로 예외가 있을 수 있으면 피한다.
- 초보자용 튜토리얼처럼 모든 기초를 풀어 쓰지 않는다.
- 실무자가 코드 리뷰에서 말해줄 법한 판단 기준을 문장으로 남긴다.

## 메타 표현과 강조 기준

요약본은 "강의가 무엇을 말했다"보다 "개념이 어떻게 동작하고, 어떤 기준으로 판단해야 하는가"를 직접 말한다.

`이날 강의는`, `강의에서`, `수업이다`, `반복된 내용은`, `중심 주제는`처럼 강의 자체를 설명하는 포장어는 줄이고, 개념과 판단 기준을 바로 주어로 둔다.

`핵심은`, `반드시`, `조심해야 한다`, `기준으로 읽어야 한다`, `버그를 만들기 쉽다` 같은 강조 표현은 필요하면 유지한다. 다만 같은 섹션 안에서 반복되거나 의미가 달라지지 않으면 판단 기준 형태의 문장으로 정리한다.

`실무에서는`, `실무 기준은`, `실무에서 조심할 점은` 같은 표현은 실무 판단 톤을 유지하고, 개념 설명에서 판단 기준으로 넘어가는 전환 역할을 할 수 있다.

HTML/CSS 속성, 태그, 값과 JavaScript API, 키워드, 연산자는 `h1`, `section`, `width: 100%`, `box-sizing`, `overflow: hidden`처럼 인라인 코드로 표시한다.

강의에서 비유적으로 설명한 말은 그대로 옮기기보다 기술 문맥에 맞게 정리한다.

## 제목 작성 기준

제목은 단순 키워드보다 그 섹션의 결론이나 관점을 드러내는 쪽이 좋다.

좋은 제목:

- `태그는 정보 구조를 만든다`
- `CSS는 CSSOM을 수정하는 언어다`
- `Margin은 border-box 밖에 있다`

덜 좋은 제목:

- `태그`
- `CSS`
- `Margin`

단, 표나 목록 성격이 강한 섹션은 `주요 인라인 태그`처럼 짧은 제목도 가능하다.

## 실무 판단과 트레이드오프

요약본은 개념 설명에서 끝나지 않고, 실무에서 어떤 판단을 해야 하는지까지 남긴다.

특히 독학 경험자는 "작동하는 코드"는 만들 수 있지만, 왜 그 코드가 안전한지 또는 위험한지 판단하는 기준이 부족할 수 있다. 요약본은 이 판단 기준을 보강하는 방향으로 쓴다.

기술 선택은 장점만 갖는 것처럼 쓰지 않는다. 왜 쓰는지, 무엇이 좋아지는지, 무엇이 불편해지는지, 어떤 기준으로 선택해야 하는지를 함께 정리한다.

예를 들어 이미지 스프라이트는 서버 요청 수를 줄이는 성능 이점과 개별 이미지 수정/교체의 유지보수 비용을 함께 본다.

이 기준은 이미지 스프라이트뿐 아니라 `overflow: hidden`, web font, background image, reset CSS처럼 장단점이 함께 있는 주제에도 적용한다.

## 목록과 표 사용 기준

비슷한 성격의 항목을 나열할 때는 목록을 사용한다.

순서가 중요한 절차는 번호 목록을 사용한다.

표는 비교 항목이 많고, 한 줄 설명으로 충분할 때만 사용한다. 설명이 길어지면 표보다 일반 문단이나 목록이 읽기 쉽다.

## 최종 점검 기준

압축이 끝난 뒤에는 다음을 확인한다.

- 첫 섹션에서 그날 강의의 큰 관점이 잡히는가
- 섹션 제목만 읽어도 흐름이 보이는가
- 마지막 섹션명이 `N일차 실무 감각` 형식으로 통일되어 있는가
- 세부 개념을 지나치게 잘게 쪼개지 않고, 함께 판단해야 하는 흐름끼리 묶었는가
- 전사체 표현이나 불필요한 반복이 남아 있지 않은가
- 보조 자료의 내용을 그대로 옮기지 않고, 개념 강조점과 판단 기준만 반영했는가
- 누락 구간이나 사용자 메모의 핵심이 빠지지 않았는가
- 코드 예시는 개념 이해에 필요한 만큼만 남았는가
- 비슷한 코드 예시가 여러 개 반복되지 않는가
- 반복 개념이 다른 날짜 파일에 의존하지 않고도 이해되는가
- 반복 개념에서 긴 재설명보다 그날의 적용 판단이 중심이 되었는가
- 중요한 트레이드오프가 누락되지 않았는가
- 실무에서 어떤 판단을 해야 하는지 문장으로 남아 있는가
- `이날 강의는`, `강의에서` 같은 메타 표현이 불필요하게 반복되지 않는가
- 강조 표현은 남기되, 강조의 대상이 강의 자체가 아니라 개념과 판단 기준으로 향하는가
- 본문이 압축 판단 자체를 설명하지 않고, 남은 개념 자체를 서술하는가
- 너무 기초적인 설명이 길게 남아 있지 않은가
- 사용자의 선행 지식을 바탕으로 새로 판단해야 할 원리와 주의점이 중심이 되었는가
- 독학자가 놓치기 쉬운 원리, 예외, 주의점이 보강되어 있는가
- 05.07~05.26과 비슷한 문체와 밀도를 유지하는가
